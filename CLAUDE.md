`dev-knowledge`: プログラミングで得た知見・遭遇した問題と解決策を蓄積する個人ナレッジベース。Docusaurus + GitHub Pages で公開する。

## 目的

- 雑多に蓄積することを優先する。整理コストで蓄積のハードルを上げない。
- 後から検索・再利用できる状態を保つ。横断はタグで担保する。

## 依存

- 新しいライブラリは基本入れない。

## ディレクトリ構成

技術ごとにフォルダを分ける。

```
docs/
  react/
  rust/
  ...（技術が増えたら追加）
```

- カテゴリ（フォルダ）は粗く保つ。細分化は記事が溜まってからでよい。
- 1 記事 = 1 つの問題 or 知見。粒度は小さく。

## 記事を書くときのルール

各 `.md` の frontmatter に以下を付ける。

```yaml
---
title: 簡潔なタイトル
tags: [react, hooks, ...]
---
```

- `tags` は横断検索の主役。技術名・トピック名を惜しまず付ける。
- 「遭遇した問題 → 原因 → 解決」の流れを基本にする。再現条件と環境（バージョン等）を残す。
- コードは最小再現に絞る。動かない例と動く例を併記すると後で役立つ。

## やらないこと

- 過剰な階層化・カテゴリの作り込み。
- 1 記事に複数トピックを詰め込むこと。

## Issues から起動されたときの動き

GitHub の Issue から起動された場合、ラベルで「調査するか・しないか」を決める。

### ラベル別の挙動

- **`knowledge:direct`**: Issue に書かれている内容をそのまま記事化してよい。追加調査はしない。完成したら `main` にそのままプッシュしてよい（PR 不要）。
- **`knowledge:pr`**: サブエージェントを使って調査させ、その調査結果を元にナレッジ記事を生成する。完成したら PR を作成する（`main` への直接プッシュはしない）。
  - 複数の観点があるときは、観点ごとにサブエージェントを並行で起動して調査させる。
  - サブエージェントの調査結果を統合し、本リポジトリの記事ルール（frontmatter・粒度・タグ）に沿って記事化する。
- **ラベルなし**: Issue の内容を見て自己判断する。
  - 基本的に、Issue 本文に記事の `.md` 内容（知見・問題と解決など）が既に貼られていれば、調査なしで記事化してよい。
  - 内容が断片的・要調査と判断したら、`knowledge:pr` と同じ流れで調査してから記事化する。

### 補足

- どのケースでも、生成する記事は上記「記事を書くときのルール」「ディレクトリ構成」に従う。
- 調査が必要なケースでは、調査の根拠（参照したバージョン・出典など）を記事に残す。

## 言語

- 記事・コメントは日本語で書く。
