## TL;DR

- 実装方式は 3 つの軸の組み合わせ: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({ onConfirm, onCancel })`）
  3. **Provider + Promise 方式**（`await confirm({...})` — material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカル）
- 「Provider 方式 = Promise ベース」とは限らない。コールバック方式は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure 等の罠を回避できる。

## 射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を、動作する完全コードで比較する。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準を得る。

## 軸の整理

混同しがちな 3 つの軸を分離する。

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の useState / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onConfirm` 等を渡す） / (β) コールバック方式（`show({ onConfirm })`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称。`Provider 方式 ≠ Promise 方式` である点に注意。

## 共通の前提コード

```ts
// src/api/items.ts
export type Item = { id: string; name: string };

export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: "DELETE" });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

以下、UI ライブラリは shadcn/ui の `AlertDialog`（Radix UI ベース）を想定する。`AlertDialog` を **controlled**（`open` / `onOpenChange`）で使うのが基本。非同期処理を待ってから閉じたい場合、`AlertDialogAction` / `AlertDialogCancel` の `onClick` で `e.preventDefault()` しないと Radix が即座に閉じてしまう点に注意（下記コードに反映済み）。

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブを提供するだけで、state は呼び出し側で `useState` 管理。

- **state**: 呼び出し側 / **API**: props 駆動 / **UI**: 利用側組み立て

```tsx
// src/patterns/A/ItemListA.tsx
import { useState } from "react";
import { type Item, deleteItem } from "@/api/items";
import {
  AlertDialog,
  AlertDialogContent,
  AlertDialogHeader,
  AlertDialogFooter,
  AlertDialogTitle,
  AlertDialogDescription,
  AlertDialogAction,
  AlertDialogCancel,
} from "@/components/ui/alert-dialog";

const initialItems: Item[] = [
  { id: "1", name: "りんご" },
  { id: "2", name: "みかん" },
];

export function ItemListA() {
  const [items, setItems] = useState<Item[]>(initialItems);
  // 削除対象。null のときダイアログは閉じている
  const [target, setTarget] = useState<Item | null>(null);
  const [isPending, setIsPending] = useState(false);

  const handleConfirm = async () => {
    if (!target) return;
    setIsPending(true);
    try {
      await deleteItem(target.id);
      setItems((prev) => prev.filter((i) => i.id !== target.id));
      setTarget(null); // 成功したら閉じる
    } catch (e) {
      console.error(e);
      // 失敗時はダイアログを開いたままにしてエラー表示する等
    } finally {
      setIsPending(false);
    }
  };

  return (
    <div>
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            {item.name}
            <button onClick={() => setTarget(item)}>削除</button>
          </li>
        ))}
      </ul>

      {/* controlled: open は target の有無、onOpenChange で閉じる操作を受ける */}
      <AlertDialog
        open={target !== null}
        onOpenChange={(open) => {
          if (!open && !isPending) setTarget(null); // 処理中は閉じさせない
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>削除しますか？</AlertDialogTitle>
            <AlertDialogDescription>
              「{target?.name}」を削除します。この操作は取り消せません。
            </AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel disabled={isPending}>キャンセル</AlertDialogCancel>
            {/* preventDefault しないと Radix が即座に閉じる。非同期を待つため自前で閉じる */}
            <AlertDialogAction
              disabled={isPending}
              onClick={(e) => {
                e.preventDefault();
                void handleConfirm();
              }}
            >
              {isPending ? "削除中…" : "削除する"}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </div>
  );
}
```

- **長所**: 依存が薄い（Radix のみ）。挙動が完全に透明で state がローカルに見える。アクセシビリティは Radix が担保。
- **短所**: 確認が必要な箇所ごとに `useState`（open / target / pending）と JSX のボイラープレートが繰り返される。「await して結果を受け取る」感覚では書けない。

## パターン B: Provider + コールバック方式

App root に `ConfirmProvider` を 1 つ置き、`showConfirm({ onConfirm, onCancel })` を呼ぶ。**Promise の resolve トリックを使わず**、`onConfirm` が返す Promise を Provider が `await` して `isPending` を一元管理する。

- **state**: Provider が保持 / **API**: コールバック / **UI**: Provider 内が提供

```tsx
// src/patterns/B/ConfirmProvider.tsx
import {
  createContext,
  useCallback,
  useContext,
  useMemo,
  useRef,
  useState,
  type ReactNode,
} from "react";
import {
  AlertDialog,
  AlertDialogContent,
  AlertDialogHeader,
  AlertDialogFooter,
  AlertDialogTitle,
  AlertDialogDescription,
  AlertDialogAction,
  AlertDialogCancel,
} from "@/components/ui/alert-dialog";

export type ConfirmOptions = {
  title?: string;
  description?: string;
  confirmText?: string;
  cancelText?: string;
  // onConfirm が Promise を返すと Provider が await して isPending を管理する
  onConfirm: () => void | Promise<void>;
  onCancel?: () => void;
};

type ConfirmContextValue = { showConfirm: (options: ConfirmOptions) => void };
const ConfirmContext = createContext<ConfirmContextValue | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [open, setOpen] = useState(false);
  const [isPending, setIsPending] = useState(false);
  const optionsRef = useRef<ConfirmOptions | null>(null);

  const showConfirm = useCallback((options: ConfirmOptions) => {
    optionsRef.current = options;
    setIsPending(false);
    setOpen(true);
  }, []);

  const close = useCallback(() => {
    setOpen(false);
    optionsRef.current = null;
  }, []);

  const handleConfirm = useCallback(async () => {
    const opts = optionsRef.current;
    if (!opts) return;
    try {
      setIsPending(true);
      // キモ: onConfirm の戻り Promise を Provider が await する
      await opts.onConfirm();
      close();
    } catch (e) {
      console.error(e);
    } finally {
      setIsPending(false);
    }
  }, [close]);

  const handleCancel = useCallback(() => {
    optionsRef.current?.onCancel?.();
    close();
  }, [close]);

  const value = useMemo<ConfirmContextValue>(() => ({ showConfirm }), [showConfirm]);
  const opts = optionsRef.current;

  return (
    <ConfirmContext.Provider value={value}>
      {children}
      <AlertDialog
        open={open}
        onOpenChange={(next) => {
          if (!next && !isPending) handleCancel();
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>{opts?.title ?? "確認"}</AlertDialogTitle>
            {opts?.description && (
              <AlertDialogDescription>{opts.description}</AlertDialogDescription>
            )}
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel
              disabled={isPending}
              onClick={(e) => {
                e.preventDefault();
                handleCancel();
              }}
            >
              {opts?.cancelText ?? "キャンセル"}
            </AlertDialogCancel>
            <AlertDialogAction
              disabled={isPending}
              onClick={(e) => {
                e.preventDefault();
                void handleConfirm();
              }}
            >
              {isPending ? "処理中…" : (opts?.confirmText ?? "OK")}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm(): ConfirmContextValue["showConfirm"] {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error("useConfirm must be used within <ConfirmProvider>");
  return ctx.showConfirm;
}
```

```tsx
// src/patterns/B/ItemListB.tsx
import { useState } from "react";
import { type Item, deleteItem } from "@/api/items";
import { useConfirm } from "./ConfirmProvider";

export function ItemListB() {
  const [items, setItems] = useState<Item[]>([
    { id: "1", name: "りんご" },
    { id: "2", name: "みかん" },
  ]);
  const showConfirm = useConfirm();

  const handleDelete = (item: Item) => {
    showConfirm({
      title: "削除しますか？",
      description: `「${item.name}」を削除します。`,
      confirmText: "削除する",
      // 戻り値の Promise を Provider が await → isPending を一元管理してくれる
      onConfirm: async () => {
        await deleteItem(item.id);
        setItems((prev) => prev.filter((i) => i.id !== item.id));
      },
    });
  };

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.name}
          <button onClick={() => handleDelete(item)}>削除</button>
        </li>
      ))}
    </ul>
  );
}
```

`<ConfirmProvider>` で `<ItemListB />` を包めば使える。

- **長所**: ダイアログの DOM / 状態が 1 箇所に集約。`isPending` を Provider が一元管理するので呼び出し側は処理本体だけ書けばよい。Promise resolve トリックの分かりにくさがない。
- **短所**: 結果が「コールバック」なので、削除後に続けて分岐するフローはコールバック内に押し込む形になる（`await confirm()` のような直線的な書き方にはならない）。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`const { confirmed, reason } = await confirm({...})` で線形に書ける。

- **state**: Provider + `useRef` で resolver 保持 / **API**: Promise / **UI**: Provider 内が提供

### C-1. 実ライブラリ material-ui-confirm v4 を使う

```tsx
// src/patterns/C/ItemListC.tsx
import { useState } from "react";
import { useConfirm } from "material-ui-confirm";
import { type Item, deleteItem } from "@/api/items";

export function ItemListC() {
  const [items, setItems] = useState<Item[]>([
    { id: "1", name: "りんご" },
    { id: "2", name: "みかん" },
  ]);
  const [pendingId, setPendingId] = useState<string | null>(null);
  // v4: useConfirm() は confirm 関数そのものを返す（{ confirm } ではない）
  const confirm = useConfirm();

  const handleDelete = async (item: Item) => {
    // v4: reject せず必ず { confirmed, reason } で解決する
    const { confirmed, reason } = await confirm({
      title: "削除しますか？",
      description: `「${item.name}」を削除します。`,
      confirmationText: "削除する",
      cancellationText: "キャンセル",
    });

    // reason: "confirm" | "cancel" | "natural" | "unmount"
    if (!confirmed) {
      console.log("cancelled:", reason);
      return;
    }

    setPendingId(item.id);
    try {
      await deleteItem(item.id);
      setItems((prev) => prev.filter((i) => i.id !== item.id));
    } finally {
      setPendingId(null);
    }
  };

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.name}
          <button disabled={pendingId === item.id} onClick={() => void handleDelete(item)}>
            {pendingId === item.id ? "削除中…" : "削除"}
          </button>
        </li>
      ))}
    </ul>
  );
}
```

App root を `<ConfirmProvider>`（`material-ui-confirm` の export）で包む。

### C-2. 同方式を自前実装する（useRef に resolver を保持、unmount で `reason:"unmount"`）

material-ui-confirm が内部でやっていることの最小再現。

```tsx
// src/patterns/C/ConfirmProviderPromise.tsx
import {
  createContext,
  useCallback,
  useContext,
  useEffect,
  useMemo,
  useRef,
  useState,
  type ReactNode,
} from "react";
import {
  AlertDialog,
  AlertDialogContent,
  AlertDialogHeader,
  AlertDialogFooter,
  AlertDialogTitle,
  AlertDialogDescription,
  AlertDialogAction,
  AlertDialogCancel,
} from "@/components/ui/alert-dialog";

export type ConfirmReason = "confirm" | "cancel" | "natural" | "unmount";
export type ConfirmResult = { confirmed: boolean; reason: ConfirmReason };
export type ConfirmDialogOptions = {
  title?: string;
  description?: string;
  confirmationText?: string;
  cancellationText?: string;
};

type ConfirmFn = (options?: ConfirmDialogOptions) => Promise<ConfirmResult>;
const ConfirmContext = createContext<ConfirmFn | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [open, setOpen] = useState(false);
  const [options, setOptions] = useState<ConfirmDialogOptions>({});
  // resolver を ref に保持するのがこの方式のキモ
  const resolverRef = useRef<((result: ConfirmResult) => void) | null>(null);

  const settle = useCallback((result: ConfirmResult) => {
    resolverRef.current?.(result);
    resolverRef.current = null;
    setOpen(false);
  }, []);

  const confirm = useCallback<ConfirmFn>((opts = {}) => {
    setOptions(opts);
    setOpen(true);
    return new Promise<ConfirmResult>((resolve) => {
      resolverRef.current = resolve;
    });
  }, []);

  // アンマウント時に未解決の Promise を reason:"unmount" で解決する（pending 永続を防ぐ）
  useEffect(() => {
    return () => {
      resolverRef.current?.({ confirmed: false, reason: "unmount" });
      resolverRef.current = null;
    };
  }, []);

  const value = useMemo(() => confirm, [confirm]);

  return (
    <ConfirmContext.Provider value={value}>
      {children}
      <AlertDialog
        open={open}
        onOpenChange={(next) => {
          if (!next) settle({ confirmed: false, reason: "natural" }); // ESC / バックドロップ
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>{options.title ?? "Are you sure?"}</AlertDialogTitle>
            {options.description && (
              <AlertDialogDescription>{options.description}</AlertDialogDescription>
            )}
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel
              onClick={(e) => {
                e.preventDefault();
                settle({ confirmed: false, reason: "cancel" });
              }}
            >
              {options.cancellationText ?? "Cancel"}
            </AlertDialogCancel>
            <AlertDialogAction
              onClick={(e) => {
                e.preventDefault();
                settle({ confirmed: true, reason: "confirm" });
              }}
            >
              {options.confirmationText ?? "Ok"}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm(): ConfirmFn {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error("useConfirm must be used within <ConfirmProvider>");
  return ctx;
}
```

- **長所**: `const { confirmed } = await confirm()` という直線的・命令的な書き味。結果に応じた後続処理を呼び出し側にそのまま書ける。`reason` で「なぜ閉じたか」を判別できる。
- **短所**: resolver を ref に持つ実装が必要（自前なら）。**unmount 時の取りこぼし対策を忘れると Promise が永久に pending** になる。標準では同時に複数のダイアログを出せない。

> material-ui-confirm は v3 まで cancel 時に Promise を reject していたが、v4 で破壊的変更が入り、reject せず常に `{ confirmed, reason }` で解決するようになった。`reason: "unmount"` は表示中にコンポーネントが unmount したケースを表す。

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、`import { confirm } from "./confirm"` だけで呼べる。内部で React 18 の `createRoot` を使い Dialog を `document.body` 直下に動的マウントする。`confirmable()` で包む。

- **state**: React tree 外（別 root が管理） / **API**: Promise / **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む

```tsx
// src/patterns/D/confirm.tsx
import {
  confirmable,
  createConfirmation,
  type ConfirmDialogProps,
} from "react-confirm";

type Props = { title: string; description: string };

// confirmable が注入する props: show / proceed / cancel / dismiss
// Response 型 = boolean（proceed(value) で resolve される値）
function ConfirmDialog({ show, proceed, title, description }: ConfirmDialogProps<Props, boolean>) {
  if (!show) return null;
  return (
    <div style={overlayStyle} role="alertdialog" aria-modal="true">
      <div style={dialogStyle}>
        <h2>{title}</h2>
        <p>{description}</p>
        <div style={{ display: "flex", gap: 8, justifyContent: "flex-end" }}>
          {/* proceed(value) が Promise を resolve する */}
          <button onClick={() => proceed(false)}>キャンセル</button>
          <button onClick={() => proceed(true)}>削除する</button>
        </div>
      </div>
    </div>
  );
}

const overlayStyle: React.CSSProperties = {
  position: "fixed",
  inset: 0,
  background: "rgba(0,0,0,0.5)",
  display: "flex",
  alignItems: "center",
  justifyContent: "center",
  zIndex: 9999,
};
const dialogStyle: React.CSSProperties = {
  background: "#fff",
  padding: 24,
  borderRadius: 8,
  minWidth: 320,
};

// createConfirmation が内部で createRoot して document.body 直下にマウントする。
// 返り値は (props) => Promise<boolean>
const confirmDialog = createConfirmation(confirmable(ConfirmDialog));

export function confirm(props: Props): Promise<boolean> {
  return confirmDialog(props);
}
```


> react-confirm の注入 props は `show / proceed / cancel / dismiss`。`proceed(value)` は resolve、`cancel(value?)` は **reject**（resolve とは別物）、`dismiss()` は resolve も reject もせず閉じる（Promise は pending のまま、外部制御 `controls` 用）。`createConfirmation(Component, unmountDelay = 1000, mountingNode?)` が `(props) => Promise<R>` を返す。

- **長所**: Provider セットアップ不要。`import { confirm }` だけでどこからでも呼べる（イベントハンドラ・ストアのミドルウェア等、React ツリー外からも）。`window.confirm()` のメンタルモデル。
- **短所**: `document.body` に別 root を `createRoot` でマウントするため、**呼び出し元の Context（テーマ・i18n 等）が伝播しない**。対策として react-confirm は `ContextAwareConfirmation` / `createReactTreeMounter` 等を提供しているが、それは実質ミニ Provider を root に置くことになる。

## パターン E: Render Hook（hook が JSX を返す）

LINE 証券のテックブログが初出、uhyo 氏が「render hooks」として記事化（命名は uhyo 氏の当時の上司）。日本ローカルで議論されるパターン。hook 内で `JSX.Element` を直接返すか `() => JSX.Element` を返す。**FC を hook 内で定義して返すと毎レンダー別の型になり再マウントするので避ける**（別記事「render hooks パターン」参照）。

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録） / **API**: 命令的 + JSX 返却 / **UI**: hook 内で JSX 構築

```tsx
// src/patterns/E/useConfirm.tsx
import { useCallback, useRef, useState, type ReactElement } from "react";
import {
  AlertDialog,
  AlertDialogContent,
  AlertDialogHeader,
  AlertDialogFooter,
  AlertDialogTitle,
  AlertDialogDescription,
  AlertDialogAction,
  AlertDialogCancel,
} from "@/components/ui/alert-dialog";

export type ConfirmDialogOptions = {
  title?: string;
  description?: string;
  confirmText?: string;
  cancelText?: string;
};

type UseConfirmReturn = {
  openConfirm: (options?: ConfirmDialogOptions) => Promise<boolean>;
  dialog: ReactElement; // これを JSX として画面に置く（FC は返さない）
};

export function useConfirm(): UseConfirmReturn {
  const [open, setOpen] = useState(false);
  const [options, setOptions] = useState<ConfirmDialogOptions>({});
  const resolverRef = useRef<((value: boolean) => void) | null>(null);

  const openConfirm = useCallback((opts: ConfirmDialogOptions = {}) => {
    setOptions(opts);
    setOpen(true);
    return new Promise<boolean>((resolve) => {
      resolverRef.current = resolve;
    });
  }, []);

  const settle = useCallback((value: boolean) => {
    resolverRef.current?.(value);
    resolverRef.current = null;
    setOpen(false);
  }, []);

  // ポイント: FC を新規定義して返さない。JSX.Element をそのまま組み立てて返す。
  // hook 内で function Dialog(){} を定義して <Dialog /> を返すと、毎レンダーで
  // 別コンポーネント型になり再マウント＝state(open) が喪失する。
  const dialog = (
    <AlertDialog
      open={open}
      onOpenChange={(next) => {
        if (!next) settle(false); // ESC / バックドロップ = キャンセル扱い
      }}
    >
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>{options.title ?? "確認"}</AlertDialogTitle>
          {options.description && (
            <AlertDialogDescription>{options.description}</AlertDialogDescription>
          )}
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel
            onClick={(e) => {
              e.preventDefault();
              settle(false);
            }}
          >
            {options.cancelText ?? "キャンセル"}
          </AlertDialogCancel>
          <AlertDialogAction
            onClick={(e) => {
              e.preventDefault();
              settle(true);
            }}
          >
            {options.confirmText ?? "OK"}
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );

  return { openConfirm, dialog };
}
```

```tsx
// src/patterns/E/ItemListE.tsx
import { useState } from "react";
import { type Item, deleteItem } from "@/api/items";
import { useConfirm } from "./useConfirm";

export function ItemListE() {
  const [items, setItems] = useState<Item[]>([
    { id: "1", name: "りんご" },
    { id: "2", name: "みかん" },
  ]);
  const [pendingId, setPendingId] = useState<string | null>(null);
  // hook が JSX(dialog) と命令的 API(openConfirm) を返す
  const { openConfirm, dialog } = useConfirm();

  const handleDelete = async (item: Item) => {
    const ok = await openConfirm({
      title: "削除しますか？",
      description: `「${item.name}」を削除します。`,
      confirmText: "削除する",
    });
    if (!ok) return;

    setPendingId(item.id);
    try {
      await deleteItem(item.id);
      setItems((prev) => prev.filter((i) => i.id !== item.id));
    } finally {
      setPendingId(null);
    }
  };

  return (
    <div>
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            {item.name}
            <button disabled={pendingId === item.id} onClick={() => void handleDelete(item)}>
              {pendingId === item.id ? "削除中…" : "削除"}
            </button>
          </li>
        ))}
      </ul>
      {/* hook が返した JSX.Element をそのまま描画。これを忘れるとダイアログが出ない */}
      {dialog}
    </div>
  );
}
```

- **長所**: Provider 不要で呼び出しコンポーネントに閉じる。`await openConfirm()` の命令的 API とローカルな状態管理を両立。
- **短所**: 利用側が `{dialog}` を描画し忘れると静かに無反応になる。複数箇所で使うと各々がダイアログ DOM を持つ（共有されない）。FC 再定義の落とし穴を理解していないと事故る。命名者自身が「流行る気配がない」と評しており、英語圏ライブラリ採用例なし。

## 全パターン比較表

| | A: プリミティブ | B: Provider+callback | C: Provider+Promise | D: Provider なし命令的 | E: Render Hook |
|---|---|---|---|---|---|
| **state の置き場** | 呼び出し側 | Provider | Provider + useRef | React tree 外 | hook 内（実体は呼び出し元） |
| **API スタイル** | props 駆動 | コールバック | Promise（await） | Promise（await） | 命令的 + JSX |
| **Provider 必要** | 不要 | 必要 | 必要 | 不要 | 不要 |
| **Context 伝播** | あり | あり | あり | **なし**（別 root） | あり |
| **pending 管理** | 呼び出し側 | Provider 一元 | 呼び出し側 / lib | 呼び出し側 | 呼び出し側 |
| **resolve 漏れリスク** | なし | なし | あり（要対策） | ライブラリが対策 | あり（要対策） |
| **呼び出し側コードの線形性** | △ | △ | ◎ | ◎ | ◎ |
| **ライブラリ採用例** | shadcn / Radix | 自前実装が多い | material-ui-confirm 等多数 | react-confirm 等 | 確認できず |

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、ツリー外からも呼びたい | D（Provider なし命令的）。ただし Context 非伝播に注意 |
| List ローカルで完結、Provider を導入したくない | E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・unmount 取りこぼし等の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。

## まとめ

- 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
- 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
- 「Promise の罠を避けたいが Provider の統一感は欲しい」要件には B が最適。
- ライブラリ採用例の多さは `C > D > A` の順。E（Render Hook）はライブラリ採用例なし。

## ライブラリ API の検証メモ

- **material-ui-confirm v4**: `useConfirm()` は confirm 関数そのものを返す。`await confirm()` は reject せず `{ confirmed, reason }` で解決。`reason` は `"confirm" | "cancel" | "natural" | "unmount"`。v3 は cancel 時 reject していた（破壊的変更）。
- **react-confirm**: `confirmable(Component)` が `show / proceed / cancel / dismiss` を注入。`proceed`=resolve, `cancel`=reject, `dismiss`=未解決クローズ。`createConfirmation` は `react-dom/client` の `createRoot` で `document.body` にマウント（React 18 対応）。別 root のため Context 非伝播、対策 export あり。
- **shadcn/ui AlertDialog**: Radix の薄いラッパ。controlled は `open` / `onOpenChange`。非同期を待つには `onClick` で `e.preventDefault()` が必要。
