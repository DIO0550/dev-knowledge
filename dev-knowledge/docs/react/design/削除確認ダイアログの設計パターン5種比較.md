## TL;DR

- 実装方式は 3 つの軸の組み合わせ: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({ onConfirm, onCancel })`）
  3. **Provider + Promise 方式**（`await confirm({...})` - material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカル）
- 「Provider 方式 = Promise ベース」とは限らない。コールバック方式は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure・concurrent rendering 等の罠を回避できる。

## このドキュメントの射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を比較。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準。

## 軸の整理

混同しがちな 3 つの軸を分離する:

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の useState / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onConfirm` 等を渡す） / (β) コールバック方式（`show({ onConfirm })`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称。**`Provider 方式 ≠ Promise 方式`** であることに注意。

## 共通の前提コード

```tsx
// types.ts
export type Item = {
  id: string;
  name: string;
};

// api.ts
export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: 'DELETE' });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

UI ライブラリは shadcn/ui の `AlertDialog` を想定（Radix UI ベース、destructive style 対応）。

---

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブを提供するだけで、state は呼び出し側で `useState` で管理。

- **state**: 呼び出し側
- **API**: props 駆動
- **UI**: 利用側組み立て

```tsx
import { useState } from 'react';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
} from '@/components/ui/alert-dialog';
import { deleteItem } from './api';
import type { Item } from './types';

export function DeletableRow({ item, onDeleted }: { item: Item; onDeleted: (id: string) => void }) {
  const [open, setOpen] = useState(false);

  return (
    <li>
      {item.name}
      <button onClick={() => setOpen(true)}>削除</button>
      <AlertDialog open={open} onOpenChange={setOpen}>
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>削除確認</AlertDialogTitle>
            <AlertDialogDescription>「{item.name}」を削除します。</AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel>キャンセル</AlertDialogCancel>
            <AlertDialogAction onClick={async () => { await deleteItem(item.id); onDeleted(item.id); }}>
              削除
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </li>
  );
}
```

- **長所**: 仕組みが透明、ローカル state と自然に連携。
- **短所**: 各 List ごとに `useState` と Dialog を書く必要。

> リスト全体で 1 つの確認 state にまとめたい場合は、別記事「List → Item 構造での削除確認 state は List 側に `target: Item | null` で持つ」の設計が有効。

## パターン B: Provider + コールバック方式

App root に `<ConfirmProvider>` を 1 つ置き、利用側は `showConfirm({ onConfirm, onCancel })` を呼ぶ。**Promise の resolve トリックを使わず、コールバックで結果を受け取る**。

- **state**: Provider が保持
- **API**: コールバック
- **UI**: ライブラリ（Provider 内）が提供

```tsx
import { createContext, useContext, useState, useCallback, type ReactNode } from 'react';

type ConfirmOptions = {
  title: string;
  description?: string;
  onConfirm: () => void | Promise<void>;
  onCancel?: () => void;
};

const ConfirmContext = createContext<(opts: ConfirmOptions) => void>(() => {});

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [opts, setOpts] = useState<ConfirmOptions | null>(null);
  const [isPending, setIsPending] = useState(false);

  const showConfirm = useCallback((o: ConfirmOptions) => setOpts(o), []);

  const handleConfirm = async () => {
    if (!opts) return;
    setIsPending(true);
    try {
      await opts.onConfirm(); // 戻り値 Promise を Provider が await → loading を一元管理
      setOpts(null);
    } finally {
      setIsPending(false);
    }
  };

  const handleCancel = () => {
    opts?.onCancel?.();
    setOpts(null);
  };

  return (
    <ConfirmContext.Provider value={showConfirm}>
      {children}
      {/* opts !== null が開閉を兼ねる。AlertDialog 等で描画 */}
      {opts && (
        <ConfirmDialog
          {...opts}
          isPending={isPending}
          onConfirm={handleConfirm}
          onCancel={handleCancel}
        />
      )}
    </ConfirmContext.Provider>
  );
}

export const useConfirm = () => useContext(ConfirmContext);
```

```tsx
// 利用側
const showConfirm = useConfirm();
showConfirm({
  title: '削除確認',
  description: '本当に削除しますか?',
  onConfirm: async () => { await deleteItem(item.id); onDeleted(item.id); },
});
```

- **長所**: Promise の resolve トリック不要、`onConfirm: async () => {}` がそのまま使える、アンマウント問題が起きにくい。`onConfirm` の戻り値 Promise を Provider が `await` する設計で loading（`isPending`）を Provider 側に一元化できる。
- **短所**: 呼び出し側コードが線形にならない（コールバックに処理を渡す形になる）。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`await confirm({...})` で線形に書ける。

- **state**: Provider + `useRef` で resolver 保持
- **API**: Promise
- **UI**: ライブラリ（Provider 内）が提供

```tsx
// 利用側 (material-ui-confirm 風)
const confirm = useConfirm();

const handleDelete = async () => {
  const { confirmed, reason } = await confirm({
    title: '削除確認',
    description: '本当に削除しますか?',
  });
  if (confirmed) {
    await deleteItem(item.id);
    onDeleted(item.id);
  }
  // reason === 'unmount' なら親アンマウントで閉じられたケース
};
```

実装の核は「`confirm()` 呼び出し時に Promise を作り、その `resolve` を `useRef` に保持。ユーザーがボタンを押したら対応する値で `resolve` する」という命令的な仕掛け。

- **長所**: 呼び出し側コードが線形（`if ((await confirm()).confirmed) {...}`）、既存ライブラリと API 一致。
- **短所**: resolver を `useRef` で保持する命令的仕掛け、resolve 漏れ・stale closure のリスク。

material-ui-confirm は **v4.0.0** で「アンマウント時に Promise が永遠に pending」問題を `{ confirmed, reason }` API で解決した（`reason` は `"confirm" | "cancel" | "natural" | "unmount"` の 4 値。親アンマウント時は `reason: "unmount"`）。v3 の旧挙動は `useLegacyReturn` で復元できる。

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、グローバル関数として `import { confirm } from './confirm'` する。内部で React 18 の `createRoot` で Dialog を `document.body` 直下に動的マウントする。

- **state**: React tree 外（別 root が管理）
- **API**: Promise
- **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む

```tsx
// confirm() を呼ぶと document.body 直下に root が作られ Dialog がマウントされる
const ok = await confirm({ title: '削除確認', description: '本当に削除しますか?' });
if (ok) await deleteItem(item.id);
```

- **長所**: Provider セットアップ不要、`window.confirm()` のメンタルモデル。
- **短所**: デフォルトでは別 root のため Context にアクセス不可。`ContextAwareConfirmation.ConfirmationRoot` をアプリの root に置けば Context 対応可能（＝実質ミニ Provider）。

react-confirm の既定マウンタは `createRoot(wrapper)` → `root.render(...)` を使い、`mountNode || defaultMountNode || document.body` に div を `appendChild` する（既定は `document.body` 直下）。

## パターン E: Render Hook（hook が JSX を返す）

LINE 証券のテックブログで紹介され、uhyo 氏が「render hooks」として記事化（命名は uhyo 氏の当時の上司）。日本ローカルで議論されるパターン。

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録）
- **API**: コールバック + JSX 返却
- **UI**: hook 内で JSX 構築

```tsx
// JSX.Element を返す形（FC を返すと再マウントするので避ける）
const [confirmDialog, openConfirm] = useConfirmDialog();
return (
  <div>
    <button onClick={() => openConfirm(() => deleteItem(item.id))}>削除</button>
    {confirmDialog /* ← JSX に置き忘れると静かに動かない */}
  </div>
);
```

- **長所**: Provider 不要、ローカルに閉じる。
- **短所**: JSX に `{confirmDialog}` を置き忘れると静かに動かない / UI が hook に内包され再利用性低下 / 命名者周辺自身が「流行る気配がない」と評している / 英語圏ライブラリ採用例なし。

**重要な落とし穴**: hook 内で `React.FC` を定義して返すと、Custom Hook 呼び出し毎に再マウントされる（fneco 氏の検証あり）。必ず `JSX.Element` を直接返すか `() => JSX.Element` を返す形式にする。詳細は別記事「render hooks パターン - FC 返しは再マウント、JSX.Element 返しが本来の形」参照。

## 全パターン比較表

| | A: プリミティブ | B: Provider + callback | C: Provider + Promise | D: Provider なし命令的 | E: Render Hook |
|---|---|---|---|---|---|
| **state の置き場** | 呼び出し側 | Provider | Provider + useRef | React tree 外 | hook 内（実体は呼び出し元） |
| **API スタイル** | props 駆動 | コールバック | Promise（await） | Promise（await） | コールバック + JSX |
| **Provider 必要** | 不要 | 必要 | 必要 | 不要 | 不要 |
| **Context アクセス** | 自然 | 自然 | 自然 | デフォルト不可 | 自然 |
| **resolve 漏れリスク** | なし | なし | あり（要対策） | ライブラリが対策 | なし |
| **await 跨ぎ stale closure** | なし | なし | あり（要規律） | あり（要規律） | なし |
| **呼び出し側コードの線形性** | △ | △ | ◎ | ◎ | △ |
| **再利用範囲** | 各 List で重複 | アプリ全体 | アプリ全体 | アプリ全体 | hook 再利用可 |
| **ライブラリ採用例** | shadcn / Radix | （自前実装が多い） | material-ui-confirm 等多数 | react-confirm 等 | 確認できず |

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、Context 不要 | D（Provider なし命令的、react-confirm 等） |
| List ローカルで完結、Provider を導入したくない | A or E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・stale closure 等の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。

## まとめ

- 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
- 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
- 「Promise の罠を避けたいが、Provider の統一感は欲しい」要件には B が最適。
- ライブラリ採用例の多さは `C > D > A` の順。E（Render Hook）はライブラリ採用例なし。
