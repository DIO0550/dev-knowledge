## TL;DR

- 「アイテム削除時に確認ダイアログを出す」実装は 3 つの軸の組み合わせで整理できる: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props 駆動 / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({ onConfirm })`）
  3. **Provider + Promise 方式**（`await confirm({...})` / material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカルで議論されるパターン）
- 「Provider 方式 = Promise ベース」とは限らない。コールバック方式（パターン B）は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure 等の罠を回避できる。
- 「Promise の罠を避けたいが Provider の統一感は欲しい」要件には B が筋がいい。`material-ui-confirm` 等の既存ライブラリを使うなら実質 C 一択。

## このドキュメントの射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を、動く完全コードで比較する。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準を整理する。

## 軸の整理

混同しがちな 3 つの軸を分離する。

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の `useState` / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onConfirm` 等を渡す） / (β) コールバック方式（`show({ onConfirm })`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称に過ぎない。**`Provider 方式 ≠ Promise 方式`** である点に注意。

## 共通の前提コード

以下、サンプル全体で使う型と削除関数。

```tsx
// types.ts
export type Item = {
  id: string;
  name: string;
};

// api.ts
export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: 'DELETE' });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

UI ライブラリは shadcn/ui の `AlertDialog`（Radix UI ベース、destructive style 対応）を想定する。

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブを提供するだけで、state は呼び出し側で `useState` で管理する。

### 実装

```tsx
// ItemList.tsx
import { useState } from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';
import { Button } from '@/components/ui/button';
import { deleteItem } from './api';
import type { Item } from './types';

type Props = {
  items: Item[];
  onDeleted: (id: string) => void;
};

export function ItemList({ items, onDeleted }: Props) {
  // 「対象 Item or null」で開閉と対象を 1 つの state にまとめる
  const [target, setTarget] = useState<Item | null>(null);
  const [isDeleting, setIsDeleting] = useState(false);

  const handleConfirm = async () => {
    if (!target) return;
    setIsDeleting(true);
    try {
      await deleteItem(target.id);
      onDeleted(target.id);
      setTarget(null);
    } finally {
      setIsDeleting(false);
    }
  };

  return (
    <>
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            {item.name}
            <Button variant="destructive" onClick={() => setTarget(item)}>
              削除
            </Button>
          </li>
        ))}
      </ul>

      <AlertDialog
        open={target !== null}
        onOpenChange={(open) => {
          if (!open && !isDeleting) setTarget(null);
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>削除確認</AlertDialogTitle>
            <AlertDialogDescription>
              「{target?.name}」を削除します。この操作は取り消せません。
            </AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel disabled={isDeleting}>キャンセル</AlertDialogCancel>
            <AlertDialogAction
              onClick={(e) => {
                e.preventDefault(); // 削除完了まで自動クローズさせない
                handleConfirm();
              }}
              disabled={isDeleting}
            >
              {isDeleting ? '削除中...' : '削除'}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </>
  );
}
```

### 特徴

- **state**: 呼び出し側 / **API**: props 駆動 / **UI**: 利用側組み立て

| 長所 | 短所 |
|---|---|
| 仕組みが透明でデバッグが容易 | 各 List ごとに `useState` と Dialog を書く必要 |
| `isDeleting` などのローカル state と自然に連携できる | 削除以外の確認 UI で同じコードを書き直す |
| 「target は Item or null」で開閉と対象を 1 つの state にまとめられる | アプリ全体で統一したい場合は冗長 |

## パターン B: Provider + コールバック方式

App root に `<ConfirmProvider>` を 1 つ置き、利用側は `showConfirm({ onConfirm, onCancel })` を呼ぶ。**Promise の resolve トリックを使わず、コールバックで結果を受け取る**。

### 実装

```tsx
// ConfirmProvider.tsx
import {
  createContext,
  useCallback,
  useContext,
  useState,
  type ReactNode,
} from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ShowConfirmOptions = {
  title?: ReactNode;
  description: ReactNode;
  confirmText?: string;
  cancelText?: string;
  destructive?: boolean;
  onConfirm: () => void | Promise<void>;
  onCancel?: () => void;
};

type ConfirmContextValue = {
  showConfirm: (options: ShowConfirmOptions) => void;
};

const ConfirmContext = createContext<ConfirmContextValue | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState<ShowConfirmOptions | null>(null);
  const [isPending, setIsPending] = useState(false);

  const showConfirm = useCallback((options: ShowConfirmOptions) => {
    setState(options);
  }, []);

  const handleConfirm = async () => {
    if (!state) return;
    setIsPending(true);
    try {
      await state.onConfirm(); // 呼び出し側が渡した Promise を Provider が await
      setState(null);
    } finally {
      setIsPending(false);
    }
  };

  const handleCancel = () => {
    if (isPending) return;
    state?.onCancel?.();
    setState(null);
  };

  return (
    <ConfirmContext.Provider value={{ showConfirm }}>
      {children}
      <AlertDialog
        open={state !== null}
        onOpenChange={(open) => {
          if (!open) handleCancel();
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            {state?.title && <AlertDialogTitle>{state.title}</AlertDialogTitle>}
            <AlertDialogDescription>{state?.description}</AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel disabled={isPending}>
              {state?.cancelText ?? 'キャンセル'}
            </AlertDialogCancel>
            <AlertDialogAction
              onClick={(e) => {
                e.preventDefault();
                handleConfirm();
              }}
              disabled={isPending}
              className={
                state?.destructive
                  ? 'bg-destructive text-destructive-foreground hover:bg-destructive/90'
                  : undefined
              }
            >
              {isPending ? '処理中...' : state?.confirmText ?? 'OK'}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm(): ConfirmContextValue {
  const ctx = useContext(ConfirmContext);
  if (!ctx) {
    throw new Error('useConfirm must be used inside ConfirmProvider');
  }
  return ctx;
}
```

```tsx
// App.tsx
import { ConfirmProvider } from './ConfirmProvider';
import { ItemListPage } from './ItemListPage';

export function App() {
  return (
    <ConfirmProvider>
      <ItemListPage />
    </ConfirmProvider>
  );
}
```

```tsx
  items: Item[];
  onDeleted: (id: string) => void;

          「<strong>{item.name}</strong>」を削除します。この操作は取り消せません。
        </>
      ),
      confirmText: '削除',

      onConfirm: async () => {
        await deleteItem(item.id);
        onDeleted(item.id);
      },

  };

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.name}
          <button onClick={() => handleDelete(item)}>削除</button>
        </li>
      ))}
    </ul>
  );

```

### 特徴

- **state**: Provider が保持 / **API**: コールバック / **UI**: ライブラリ（Provider 内）が提供

| 長所 | 短所 |
|---|---|
| Promise の resolve トリック不要 → ref 不要、resolve 漏れなし | 呼び出し側コードが線形にならない（コールバックの中に処理を書く） |
| `onConfirm: async () => {}` がそのまま使える | 「確認後に複数ステップ続く」場合にコールバックがネストする |
| アンマウント問題が起きにくい（hanging Promise が無い） | Promise を期待する API（Suspense 等）と直接組み合わせにくい |
| concurrent rendering と相性が良い（state 駆動のみ） | |

### Promise との関係

`onConfirm: async () => {}` の戻り値 Promise を Provider が `await` する設計にしているので、削除中の loading 状態（`isPending`）を Provider 側で一元管理できる。**「await すべき Promise」を作るのは呼び出し側、「await する」のは Provider** なので、Promise が宙に浮く問題が起きない。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`await confirm({...})` で線形に書ける。

### 実装

```tsx
// ConfirmProvider.tsx
import {
  createContext,
  useCallback,
  useContext,
  useRef,
  useState,
  type ReactNode,
} from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ConfirmOptions = {
  title?: ReactNode;
  description: ReactNode;
  confirmText?: string;
  cancelText?: string;
  destructive?: boolean;
};

type ConfirmState = ConfirmOptions & { open: boolean };

const initialState: ConfirmState = { open: false, description: '' };

type ConfirmFn = (options: ConfirmOptions) => Promise<boolean>;

const ConfirmContext = createContext<ConfirmFn | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState<ConfirmState>(initialState);
  const resolverRef = useRef<((confirmed: boolean) => void) | null>(null);

  const confirm = useCallback<ConfirmFn>((options) => {
    return new Promise<boolean>((resolve) => {
      // 前回の Promise が残っている場合は false で resolve（hanging を防ぐ）
      resolverRef.current?.(false);
      resolverRef.current = resolve;
      setState({ ...options, open: true });
    });
  }, []);

  const close = (confirmed: boolean) => {
    resolverRef.current?.(confirmed);
    resolverRef.current = null;
    setState((prev) => ({ ...prev, open: false }));
  };

  return (
    <ConfirmContext.Provider value={confirm}>
      {children}
      <AlertDialog
        open={state.open}
        onOpenChange={(open) => {
          if (!open) close(false);
        }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            {state.title && <AlertDialogTitle>{state.title}</AlertDialogTitle>}
            <AlertDialogDescription>{state.description}</AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel onClick={() => close(false)}>
              {state.cancelText ?? 'キャンセル'}
            </AlertDialogCancel>
            <AlertDialogAction
              onClick={() => close(true)}
              className={
                state.destructive
                  ? 'bg-destructive text-destructive-foreground hover:bg-destructive/90'
                  : undefined
              }
            >
              {state.confirmText ?? 'OK'}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm(): ConfirmFn {
  const ctx = useContext(ConfirmContext);
  if (!ctx) {
    throw new Error('useConfirm must be used inside ConfirmProvider');
  }
  return ctx;
}
```

```tsx
// ItemList.tsx
import { useState } from 'react';
import { useConfirm } from './ConfirmProvider';
import { deleteItem } from './api';
import type { Item } from './types';

type Props = {
  items: Item[];
  onDeleted: (id: string) => void;
};

export function ItemList({ items, onDeleted }: Props) {
  const confirm = useConfirm();
  const [deletingId, setDeletingId] = useState<string | null>(null);

  const handleDelete = async (item: Item) => {
    const ok = await confirm({
      title: '削除確認',
      description: (
        <>
          「<strong>{item.name}</strong>」を削除します。この操作は取り消せません。
        </>
      ),
      confirmText: '削除',
      destructive: true,
    });
    if (!ok) return;

    // loading state は Provider が await していないので呼び出し側で持つ
    setDeletingId(item.id);
    try {
      await deleteItem(item.id);
      onDeleted(item.id);
    } finally {
      setDeletingId(null);
    }
  };

  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.name}
          <button
            onClick={() => handleDelete(item)}
            disabled={deletingId === item.id}
          >
            {deletingId === item.id ? '削除中...' : '削除'}
          </button>
        </li>
      ))}
    </ul>
  );
}
```

### 特徴

- **state**: Provider が保持（+ `useRef` で resolver を保持） / **API**: Promise / **UI**: ライブラリ提供

| 長所 | 短所 |
|---|---|
| 呼び出し側コードが線形（`if (await confirm()) { ... }`） | resolver を `useRef` で保持する命令的な仕掛けが必要 |
| 削除以外の確認 UI を統一できる | Promise が宙に浮く可能性（アンマウント・二重呼び出し） |
| 既存ライブラリ（material-ui-confirm 等）の API と一致 | `await` 跨ぎで `item` を握り続ける stale closure に注意 |
| | 削除中の loading state は呼び出し側で別途持つ必要（B と違い Provider が `await` していない） |

#### hanging Promise への対策（material-ui-confirm v4）

`material-ui-confirm` v4 では、v3 までの「confirm で resolve / cancel で reject」をやめ、**Promise は常に resolve される**設計に変更された。戻り値の型は次の通りで、backdrop クリックや Escape による閉鎖（`natural`）、親のアンマウント（`unmount`）でも固有の `reason` 付きで必ず resolve される（一次情報: 後述の v4 CHANGELOG / 型定義で確認）。

```ts
// material-ui-confirm v4 の confirm() 戻り値
type ConfirmResult = {
  confirmed: boolean;
  reason: 'confirm' | 'cancel' | 'natural' | 'unmount';
};
// useConfirm: () => (options?: ConfirmOptions) => Promise<ConfirmResult>
```

CHANGELOG では「コールバックが必ず最終的に実行されることが保証され、リソース解放のリーク回避に役立つ」と説明されている。自前実装する場合も、上の `close(false)` のように「ダイアログを閉じる経路すべてで必ず resolve する」配慮が要る。

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、`import { confirm } from './confirm'` のようにモジュール関数として使う。内部で React 18 の `createRoot` を使い、Dialog を `document.body` 直下に動的マウントする。

### 実装（react-confirm を使う場合）

```tsx
// MyConfirmDialog.tsx
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';
import type { ConfirmDialogProps } from 'react-confirm';
import type { ReactNode } from 'react';

type Params = {
  title?: ReactNode;
  description: ReactNode;
  confirmText?: string;
  cancelText?: string;
  destructive?: boolean;
};

// confirmable が show / proceed などの props を注入する
export const MyConfirmDialog = ({
  show,
  proceed,
  title,
  description,
  confirmText = 'OK',
  cancelText = 'キャンセル',
  destructive,
}: ConfirmDialogProps<Params, boolean>) => (
  <AlertDialog
    open={show}
    onOpenChange={(open) => {
      if (!open) proceed(false);
    }}
  >
    <AlertDialogContent>
      <AlertDialogHeader>
        {title && <AlertDialogTitle>{title}</AlertDialogTitle>}
        <AlertDialogDescription>{description}</AlertDialogDescription>
      </AlertDialogHeader>
      <AlertDialogFooter>
        <AlertDialogCancel onClick={() => proceed(false)}>
          {cancelText}
        </AlertDialogCancel>
        <AlertDialogAction
          onClick={() => proceed(true)}
          className={
            destructive
              ? 'bg-destructive text-destructive-foreground hover:bg-destructive/90'
              : undefined
          }
        >
          {confirmText}
        </AlertDialogAction>
      </AlertDialogFooter>
    </AlertDialogContent>
  </AlertDialog>
);
```

App root には何も追加しない。`confirm` は import するだけで使える。

### 特徴

- **state**: React tree 外（`createRoot` で動的マウントされた別 root が管理） / **API**: Promise / **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む

| 長所 | 短所 |
|---|---|
| Provider セットアップ不要 | デフォルトでは **Context にアクセスできない**（別 root のため） |
| `import { confirm } from './confirm'` だけで使える | Context が必要なら `ContextAwareConfirmation` + `ConfirmationRoot` を root に置く必要（実質ミニ Provider） |
| `window.confirm()` のメンタルモデルに近い | 別 root にマウントするため React DevTools 上で見えにくい |
| | テストで Dialog が別 tree にマウントされる癖がある |

Context が必要なら `ContextAwareConfirmation` 版を使う（`ConfirmationRoot` を Provider 配下に置くと、別 root ではなく React ツリー内にマウントされ Context が届く）。

```tsx
// confirm.ts (Context Aware 版)
import { confirmable, ContextAwareConfirmation } from 'react-confirm';
import { MyConfirmDialog } from './MyConfirmDialog';

export const confirm = ContextAwareConfirmation.createConfirmation(
  confirmable(MyConfirmDialog),
);

// App.tsx
import { ContextAwareConfirmation } from 'react-confirm';
import { ThemeProvider } from './ThemeProvider';

export function App() {
  return (
    <ThemeProvider>
      <ItemListPage />
      <ContextAwareConfirmation.ConfirmationRoot />
    </ThemeProvider>
  );
}
```

## パターン E: Render Hook（hook が JSX を返す）

LINE 証券のテックブログが初出、uhyo 氏が「render hooks」として記事化したパターン（命名は uhyo 氏の当時の上司による）。日本ローカルで議論される。

### 実装（JSX.Element 返し版）

```tsx
// useConfirm.tsx
import { useState, type ReactNode } from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ConfirmOptions<T> = {
  title?: ReactNode;
  renderDescription: (target: T) => ReactNode;
  confirmText?: string;
  cancelText?: string;
  destructive?: boolean;
  onConfirm: (target: T) => void | Promise<void>;
};

export function useConfirm<T>(options: ConfirmOptions<T>) {
  const [target, setTarget] = useState<T | null>(null);
  const [isPending, setIsPending] = useState(false);

  const openConfirm = (item: T) => setTarget(item);

  const handleConfirm = async () => {
    if (target === null) return;
    setIsPending(true);
    try {
      await options.onConfirm(target);
      setTarget(null);
    } finally {
      setIsPending(false);
    }
  };

  const handleCancel = () => {
    if (isPending) return;
    setTarget(null);
  };

  // React.FC ではなく JSX.Element を返す（後述の落とし穴を回避）
  const dialog = (
    <AlertDialog
      open={target !== null}
      onOpenChange={(open) => {
        if (!open) handleCancel();
      }}
    >
      <AlertDialogContent>
        <AlertDialogHeader>
          {options.title && <AlertDialogTitle>{options.title}</AlertDialogTitle>}
          <AlertDialogDescription>
            {target !== null && options.renderDescription(target)}
          </AlertDialogDescription>
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel disabled={isPending}>
            {options.cancelText ?? 'キャンセル'}
          </AlertDialogCancel>
          <AlertDialogAction
            onClick={(e) => {
              e.preventDefault();
              handleConfirm();
            }}
            disabled={isPending}
          >
            {isPending ? '処理中...' : options.confirmText ?? 'OK'}
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );

  return [dialog, openConfirm] as const;
}
```

```tsx
// ItemList.tsx
import { useConfirm } from './useConfirm';
import { deleteItem } from './api';
import type { Item } from './types';

type Props = {
  items: Item[];
  onDeleted: (id: string) => void;
};

export function ItemList({ items, onDeleted }: Props) {
  const [confirmDialog, openConfirm] = useConfirm<Item>({
    title: '削除確認',
    renderDescription: (item) => (
      <>
        「<strong>{item.name}</strong>」を削除します。この操作は取り消せません。
      </>
    ),
    confirmText: '削除',
    destructive: true,
    onConfirm: async (item) => {
      await deleteItem(item.id);
      onDeleted(item.id);
    },
  });

  return (
    <>
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            {item.name}
            <button onClick={() => openConfirm(item)}>削除</button>
          </li>
        ))}
      </ul>
      {confirmDialog}
    </>
  );
}
```

### 特徴

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録） / **API**: コールバック + JSX 返却 / **UI**: hook 内で JSX 構築

| 長所 | 短所 |
|---|---|
| Provider 不要、ローカルに閉じる | JSX に `{confirmDialog}` を置き忘れると静かに動かない |
| state と UI の凝集性が高い | UI が hook に内包され、UI とロジックを別々に再利用しにくい |
| | 命名者の uhyo 氏自身が「あまり流行る気配がありません」と評している |
| | 英語圏では「render hooks」という名称での主要ライブラリ採用例は確認できず（後述） |

**重要な落とし穴**: hook 内で `React.FC` を定義して返すと、hook の state が変わるたびに別の関数オブジェクトになり、React がそれを別コンポーネントとみなしてアンマウント・再マウントを起こす（内部 state が消える）。これを避けるため、**`JSX.Element` を直接返す**か、**`() => JSX.Element` を返して呼び出し側で `{render()}` する**形にする（uhyo 氏の記事は両形式を提示。後者は拡張性が高いとされる）。

| | A: プリミティブ | B: Provider + callback | C: Provider + Promise | D: Provider なし命令的 | E: Render Hook |
|---|---|---|---|---|---|
| **state の置き場** | 呼び出し側 | Provider | Provider + useRef | React tree 外 | hook 内（実体は呼び出し元） |
| **API スタイル** | props 駆動 | コールバック | Promise（await） | Promise（await） | コールバック + JSX |
| **Provider 必要** | 不要 | 必要 | 必要 | 不要 | 不要 |
| **Context アクセス** | 自然 | 自然 | 自然 | デフォルト不可 | 自然 |
| **resolve 漏れリスク** | なし | なし | あり（要対策） | ライブラリが対策 | なし |

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、Context 不要 | D（Provider なし命令的、react-confirm 等） |
| List ローカルで完結、Provider を導入したくない | A or E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・stale closure 等の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。コールバックは「処理を引数で渡す」だけのシンプルな仕組みで、Provider 側が `await` する設計にすれば loading 状態も Provider で一元管理できる。

`material-ui-confirm` 等の既存ライブラリを使う場合は実質 C 一択になるが、自前実装するなら B を選ぶ価値が十分にある。

## まとめ

- 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
- 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
- 「Promise の罠（resolve 漏れ、await 跨ぎクロージャ）を避けたいが、Provider の統一感は欲しい」という要件には B が最適。
- ライブラリ採用例の多さは `C > D > A` の順。E（Render Hook）はライブラリ採用例なし。
- Render Hook は英語圏で「render hooks」という名称での採用例が確認できず（ただし「hook がコンポーネント/JSX を返す」類似パターン自体は Michael Jackson らが別名で議論しており皆無ではない）、命名者の uhyo 氏自身が「流行る気配がない」と評している。

## 調査の根拠（参照したバージョン・出典）

本記事の各ライブラリ・パターンに関する事実は、以下の一次情報で確認した（調査時点）。

- **react-confirm**: `haradakunihiko/react-confirm` の現行ソースを確認。`createRoot`（React 18）で `document.body` 直下にマウント（`src/mounter/domTree.tsx`）。`confirmable` / `createConfirmation` / `ContextAwareConfirmation`（`ConfirmationRoot`・`createConfirmation` を持つ）/ `proceed` / `show` props はすべて現行 API に実在。`confirm` はライブラリが直接 export するのではなく `createConfirmation` で生成してユーザーが export する。
- **material-ui-confirm v4**: 戻り値は `Promise<{ confirmed: boolean; reason: 'confirm' | 'cancel' | 'natural' | 'unmount' }>`。v3 の resolve/reject をやめ常に resolve する設計に変更（CHANGELOG v4.0.0・`src/index.d.ts`）。`ConfirmProvider` + `useConfirm()` で使用。
- **Render Hook**: 命名は uhyo 氏の当時の上司、初出は LINE 証券テックブログ「コンポーネントをカスタムフックで提供してみた」、uhyo 氏が Qiita で記事化し「あまり流行る気配がありません」と評している（uhyo 氏 Qiita 記事）。`React.FC` を hook 内で返すと再マウントが起きる問題と「`React.FC` ではなく JSX.Element / 関数を返す」解決策は、株式会社アーティスのブログ記事で詳述。
- **shadcn/ui・Radix AlertDialog**: shadcn/ui の AlertDialog は Radix UI ベースの accessible プリミティブで、Root は controlled（`open` / `onOpenChange`）対応。destructive アクション（削除確認）用途を公式が想定（shadcn/ui「Destructive」セクション）。
