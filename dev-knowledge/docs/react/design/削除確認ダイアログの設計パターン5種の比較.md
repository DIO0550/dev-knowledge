## TL;DR

- 確認ダイアログの実装方式は 3 つの軸の組み合わせで整理できる: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({ onConfirm, onCancel })`）
  3. **Provider + Promise 方式**（`await confirm({...})`、material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカルで議論されるパターン）
- 「**Provider 方式 = Promise ベース**」とは限らない。コールバック方式（パターン B）は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure・アンマウント時の永久 pending・concurrent rendering との相性問題を**原理的に回避**できる。
- Promise 方式（パターン C）の「`await confirm()` で線形に書ける」書き味は魅力だが、その代償として上記の罠を引き受ける必要がある。実例として material-ui-confirm は v4 で「常に resolve し `{ confirmed, reason }` を返す」設計へ破壊的変更し、これらの罠を解消した。

## このドキュメントの射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を、最小実装コード付きで比較する。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準をまとめる。

## 軸の整理

混同しがちな 3 つの軸を分離する。

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の `useState` / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onOpenChange` 等を渡す） / (β) コールバック方式（`show({ onConfirm })`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称にすぎない。**`Provider 方式 ≠ Promise 方式`** である点に注意。

## 共通の前提コード

以下、サンプル全体で使う型と削除関数。

```tsx
// types.ts
export type Item = {
  id: string;
  name: string;
};

// api.ts
export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: "DELETE" });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブ（`role="alertdialog"`、フォーカストラップ、Esc 対応）を提供するだけで、開閉 state は呼び出し側が `useState` で管理する宣言的スタイル。

- **state**: 呼び出し側
- **API**: props 駆動（`open` / `onOpenChange`）
- **UI**: 利用側組み立て

```tsx
import { useState } from "react";
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from "@/components/ui/alert-dialog";
import { Button } from "@/components/ui/button";

export function DeleteButton({ onConfirm }: { onConfirm: () => void }) {
  const [open, setOpen] = useState(false);

  return (
    <>
      <Button variant="destructive" onClick={() => setOpen(true)}>
        削除
      </Button>

      <AlertDialog open={open} onOpenChange={setOpen}>
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>本当に削除しますか？</AlertDialogTitle>
            <AlertDialogDescription>この操作は取り消せません。</AlertDialogDescription>
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel>キャンセル</AlertDialogCancel>
            <AlertDialogAction
              onClick={() => {
                onConfirm();
                setOpen(false);
              }}
            >
              削除
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </>
  );
}
```

- **長所**: a11y（`role=alertdialog`・フォーカストラップ・Esc・スクリーンリーダー対応）がプリミティブ側で担保される。state が明示的で追いやすい。shadcn はコードを自プロジェクトにコピーするので完全カスタマイズ可。
- **短所**: 命令的な `await confirm()` ではない。結果を Promise で受け取れないため、削除対象 id の保持や `open` 管理を呼び出し側で書く必要があり、リスト内の多数項目では state 管理が冗長になりがち。
- **落とし穴**:
  - `AlertDialogAction` / `AlertDialogCancel` はデフォルトでクリック時にダイアログを閉じる。非同期削除を待ってから閉じたい場合は `open` を自前制御し `event.preventDefault()` で自動クローズを抑止する。
  - リスト項目ごとに別ダイアログを置くと DOM が増える。1 つのダイアログを使い回し「どの項目か」を別 state で持つのが定石。

> 現在の Radix の推奨 import は `import { AlertDialog } from "radix-ui"`（単一パッケージ）。旧来の `@radix-ui/react-alert-dialog` も併存する。

## パターン B: Provider + コールバック方式

App root に `Provider` を 1 つ置き、利用側は `showConfirm({ onConfirm, onCancel })` を呼ぶ。**Promise の resolve トリックを使わず、コールバックで結果を受け取る**。

- **state**: Provider が保持
- **API**: コールバック
- **UI**: ライブラリ（Provider 内）が提供

```tsx
// CallbackConfirmProvider.tsx
import {
  createContext,
  useCallback,
  useContext,
  useState,
  type ReactNode,
} from "react";

type ShowConfirmOptions = {
  title: string;
  description?: string;
  // 戻り値が Promise なら Provider が await して pending を表示できる
  onConfirm: () => void | Promise<void>;
  onCancel?: () => void;
};

const ConfirmContext = createContext<((o: ShowConfirmOptions) => void) | null>(null);

export function CallbackConfirmProvider({ children }: { children: ReactNode }) {
  const [state, setState] = useState<ShowConfirmOptions | null>(null);
  const [isPending, setIsPending] = useState(false);

  const showConfirm = useCallback((options: ShowConfirmOptions) => {
    setState(options);
    setIsPending(false);
  }, []);

  const handleConfirm = useCallback(async () => {
    if (!state) return;
    try {
      setIsPending(true);
      // onConfirm の戻り Promise を Provider が一元的に await する
      await state.onConfirm();
      setState(null); // 成功時のみ閉じる
    } catch (e) {
      console.error(e); // エラー時はダイアログを開いたままにもできる
    } finally {
      setIsPending(false);
    }
  }, [state]);

  const handleCancel = useCallback(() => {
    state?.onCancel?.();
    setState(null);
    setIsPending(false);
  }, [state]);

  return (
    <ConfirmContext.Provider value={showConfirm}>
      {children}
      {state && (
        <div role="dialog" aria-modal className="overlay">
          <div className="dialog">
            <h2>{state.title}</h2>
            {state.description && <p>{state.description}</p>}
            <button onClick={handleCancel} disabled={isPending}>
              キャンセル
            </button>
            <button onClick={handleConfirm} disabled={isPending}>
              {isPending ? "処理中…" : "OK"}
            </button>
          </div>
        </div>
      )}
    </ConfirmContext.Provider>
  );
}

export function useConfirm() {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error("useConfirm must be used within CallbackConfirmProvider");
  return ctx;
}
```

```tsx
// 使用例
function DeleteButton({ id }: { id: string }) {
  const showConfirm = useConfirm();

  return (
    <button
      onClick={() =>
        showConfirm({
          title: "削除しますか？",
          description: "この操作は取り消せません。",
          onConfirm: async () => {
            await deleteItem(id); // await 中は Provider が isPending=true でボタン無効化
          },
          onCancel: () => console.log("キャンセル"),
        })
      }
    >
      削除
    </button>
  );
}
```

- **長所**: Promise の resolve トリック不要。`onConfirm: async () => {}` がそのまま使え、その戻り Promise を Provider が `await` するので **`isPending` を 1 箇所で一元管理**できる（二度押し防止・「処理中…」表示の共通化）。アンマウント問題が起きにくい。
- **短所**: 呼び出し側コードが線形にならない（分岐がコールバックに散る）。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`await confirm({...})` で線形に書ける。resolver を `useRef` に保持するのが Promise トリックの肝。

- **state**: Provider + `useRef` で resolver 保持
- **API**: Promise
- **UI**: ライブラリ（Provider 内）が提供

```tsx
// ConfirmProvider.tsx
import {
  createContext,
  useCallback,
  useContext,
  useRef,
  useState,
  type ReactNode,
} from "react";

type ConfirmOptions = { title: string; description?: string };
type ConfirmContextValue = (options: ConfirmOptions) => Promise<boolean>;

const ConfirmContext = createContext<ConfirmContextValue | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [open, setOpen] = useState(false);
  const [options, setOptions] = useState<ConfirmOptions | null>(null);
  // resolver を ref に保持する（再レンダーで作り直さない）
  const resolverRef = useRef<((value: boolean) => void) | null>(null);

  const confirm = useCallback((opts: ConfirmOptions) => {
    setOptions(opts);
    setOpen(true);
    return new Promise<boolean>((resolve) => {
      resolverRef.current = resolve;
    });
  }, []);

  // 罠対策: すべての閉じ経路を単一の close に集約し、必ず resolve を一回通す
  const close = useCallback((result: boolean) => {
    setOpen(false);
    resolverRef.current?.(result);
    resolverRef.current = null; // 二重 resolve / リーク防止
  }, []);

  return (
    <ConfirmContext.Provider value={confirm}>
      {children}
      {open && options && (
        <div role="dialog" aria-modal className="overlay">
          <div className="dialog">
            <h2>{options.title}</h2>
            {options.description && <p>{options.description}</p>}
            <button onClick={() => close(false)}>キャンセル</button>
            <button onClick={() => close(true)}>OK</button>
            {/* 背景クリック / Esc も必ず close(false) を通すこと */}
          </div>
        </div>
      )}
    </ConfirmContext.Provider>
  );
}

export function useConfirm() {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error("useConfirm must be used within ConfirmProvider");
  return ctx;
}
```

```tsx
// 使用例（呼び出し側が線形に書ける）
function DeleteButton({ id }: { id: string }) {
  const confirm = useConfirm();

  const handleDelete = async () => {
    const ok = await confirm({
      title: "削除しますか？",
      description: "この操作は取り消せません。",
    });
    if (!ok) return;
    await deleteItem(id); // await を跨いだ後のコードに stale closure の罠が潜む
  };

  return <button onClick={handleDelete}>削除</button>;
}
```

- **長所**: 呼び出し側コードが線形（`if (await confirm()) {...}`）。既存ライブラリと API が一致。
- **短所**: resolver を `useRef` で保持する命令的仕掛け。次節の罠を引き受ける必要がある。

### Promise 方式の 4 つの罠

この方式は「Provider 内に**可変な resolver ref という外部状態**を持つ」ため、いくつかの罠が付随する。

1. **resolve 漏れ（Promise が永久 pending）**
   OK / キャンセル以外の閉じ方（背景クリック・**Esc**・外部からの `setOpen(false)`）で resolver を呼び忘れると、`await confirm()` が永久に pending になり後続コードが固まる。これはまさに material-ui-confirm の Issue #25「Neutrally closing the dialog (e.g. when hitting Esc) causes code to hang」で報告された現象。**対策**: すべての閉じ経路を単一の `close(result)` に集約し、必ず resolver を呼んでから `null` にする（上記コードの `close`）。

2. **stale closure（await を跨いで古い変数を参照）**
   `await confirm()` は実行を中断・再開する。再開後に、中断前にキャプチャした props/state を参照していると、その間に再レンダーされても**古い値を見続ける**。原因は Promise 特有ではなくクロージャの基本挙動だが、await を挟むと「キャプチャから使用までの時間」が伸びて顕在化しやすい。**対策**: await 後に必要な最新値は `useRef` 経由で読むか、`confirm` 呼び出し時に確定値を渡し切る。

3. **アンマウント時に永久 pending**
   待機中に呼び出し元コンポーネントがルーター遷移などでアンマウントされると、resolver を呼ぶ UI が消え Promise が解決されない。**対策**: アンマウント時に `close(false)` 相当で必ず解決するか、後述の material-ui-confirm v4 のように reject せず reason 付きで必ず resolve する。

4. **concurrent rendering / Strict Mode との相性**
   React 18 の Strict Mode は開発時に「マウント→アンマウント→再マウント」を行い effect を 2 回呼ぶため、上記 3 の問題を開発中に前倒しで露出させる。また concurrent rendering ではレンダーが中断・破棄・再実行されうるので、**Promise 生成や resolver 代入をレンダー中に行うと二重実行・取りこぼしの恐れ**がある。必ずイベントハンドラ内に閉じ込めること（上記 `confirm` は `onClick` 経由でしか呼ばれない）。

### material-ui-confirm がどう解決したか

採用例が最も多い material-ui-confirm は、まさにこれらの罠への対処の歴史を持つ。

- **v3.0.13**: 呼び出し元コンポーネントがアンマウントされたらダイアログを自動で閉じる機能を追加。
- **v3 までの根本問題**: 返り値 Promise が「confirm で resolve / cancel で **reject** / ニュートラルに閉じると **pending のまま**」という設計で、pending 放置・未捕捉 reject が起きやすかった。
- **v4.0.0（破壊的変更）**: **常に Promise を resolve** するよう変更し、`{ confirmed: boolean, reason }` を返すようにした。`reason` は `"confirm" | "cancel" | "natural"（背景/Esc） | "unmount"（親アンマウント）`。これで「キャンセルで reject」「ニュートラル/アンマウントで pending」の落とし穴を解消した。

```tsx
// App.tsx
import { ConfirmProvider } from "material-ui-confirm";

export default function App() {
  return (
    <ConfirmProvider>
      <Item />
    </ConfirmProvider>
  );
}

// Item.tsx
import { useConfirm } from "material-ui-confirm";

function Item() {
  const confirm = useConfirm();

  const handleDelete = async () => {
    const { confirmed, reason } = await confirm({
      title: "削除しますか？",
      description: "この操作は取り消せません。",
      confirmationText: "削除",
      cancellationText: "キャンセル",
    });
    if (confirmed) {
      // 削除処理
    } else {
      console.log("中断:", reason); // "cancel" | "natural" | "unmount"
    }
  };

  return <button onClick={handleDelete}>削除</button>;
}
```

v3 互換が必要なら `<ConfirmProvider useLegacyReturn>`（confirm で resolve / cancel で reject / 自然閉じで pending）で旧挙動に戻せる。**移行時の注意**: v3 の「cancel で reject」を前提にした try/catch コードは v4 では catch に入らなくなる。

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、グローバル関数として `import { confirm } from "./confirm"`。内部で React 18 の `createRoot` により Dialog を新しい DOM ノードへ動的マウントする。`window.confirm()` のメンタルモデル。

- **state**: React tree 外（別 root が管理）
- **API**: Promise
- **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む

```tsx
// confirm.ts
import { confirmable, createConfirmation, type ConfirmDialogProps } from "react-confirm";

type Props = { message: string };

const MyDialog = ({ show, proceed, message }: ConfirmDialogProps<Props, boolean>) => (
  <div className={show ? "dialog show" : "dialog hide"}>
    <p>{message}</p>
    <button onClick={() => proceed(false)}>キャンセル</button>
    <button onClick={() => proceed(true)}>削除</button>
  </div>
);

export const confirm = createConfirmation(confirmable(MyDialog));

// 呼び出し側
import { confirm } from "./confirm";

const handleDelete = async () => {
  const ok = await confirm({ message: "本当に削除しますか？" });
  if (ok) {
    // 削除処理
  }
};
```

`confirmable()` が Dialog に注入する props: `show`（表示状態）、`proceed(value)`（解決して閉じる）、`dismiss()`（却下＝reject）、`cancel(value?)`（キャンセル）。

- **長所**: Provider セットアップ不要で `await confirm(...)` の命令的呼び出しができる。UI ライブラリ非依存（見た目は完全に自前）。
- **短所**: スタイル・a11y（フォーカストラップ等）は自前実装が必要。
- **落とし穴**: **デフォルトでは Context にアクセスできない**。独立 root にマウントされるため、上位の `ThemeProvider` / 認証 Context などに `useContext` で届かない。回避するには `ContextAwareConfirmation` を使い、アプリのツリー内（Provider 配下）に `ContextAwareConfirmation.ConfirmationRoot` を置く（実質ミニ Provider）。

```tsx
import { ContextAwareConfirmation } from "react-confirm";

export const confirm = ContextAwareConfirmation.createConfirmation(confirmable(MyDialog));

// App のツリー内（Provider の配下）に置く
<ThemeProvider>
  <ContextAwareConfirmation.ConfirmationRoot />
  <YourApp />
</ThemeProvider>;
```

## パターン E: Render Hook（hook が JSX を返す）

hook が「画面に置く JSX」と「トリガー関数」の両方を返す。LINE 証券のテックブログ（2020、著者は uhyo 氏本人）が初出で、uhyo 氏が後に「render hooks」として記事化したパターン。命名は当時の上司によるもので、uhyo 氏自身が「あまり流行る気配がない」と評している。**日本語ソースに議論が偏在**しており、英語圏ではむしろ「custom hook に JSX を入れるな」という否定的論調が主流。

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録）
- **API**: Promise + JSX 返却
- **UI**: hook 内で JSX 構築

```tsx
import { useCallback, useRef, useState } from "react";

type ConfirmOptions = { title?: string; description: string };

type UseConfirmReturn = {
  confirmDialog: JSX.Element; // 画面に置く JSX（直接 JSX.Element を返す）
  confirm: (options: ConfirmOptions) => Promise<boolean>;
};

export function useConfirm(): UseConfirmReturn {
  const [isOpen, setIsOpen] = useState(false);
  const [options, setOptions] = useState<ConfirmOptions | null>(null);
  const resolveRef = useRef<((value: boolean) => void) | null>(null);

  const confirm = useCallback((opts: ConfirmOptions) => {
    setOptions(opts);
    setIsOpen(true);
    return new Promise<boolean>((resolve) => {
      resolveRef.current = resolve;
    });
  }, []);

  const close = useCallback((result: boolean) => {
    setIsOpen(false);
    resolveRef.current?.(result);
    resolveRef.current = null;
  }, []);

  // ★ コンポーネント(React.FC)を定義して返さず、JSX.Element を直接返す
  const confirmDialog = isOpen ? (
    <div role="dialog" aria-modal className="confirm-overlay">
      <div className="confirm-box">
        {options?.title && <h2>{options.title}</h2>}
        <p>{options?.description}</p>
        <button onClick={() => close(false)}>キャンセル</button>
        <button onClick={() => close(true)}>OK</button>
      </div>
    </div>
  ) : (
    <></>
  );

  return { confirmDialog, confirm };
}
```

```tsx
// 使用例: {confirmDialog} を置き忘れると静かに動かない
function DeleteButton() {
  const { confirmDialog, confirm } = useConfirm();

  const handleClick = async () => {
    const ok = await confirm({ description: "本当に削除しますか？" });
    if (!ok) return;
    // 削除処理
  };

  return (
    <>
      <button onClick={handleClick}>削除</button>
      {confirmDialog} {/* ← これを忘れるとダイアログが描画されず confirm が解決しない */}
    </>
  );
}
```

- **長所**: Provider 不要でローカルに閉じる。
- **短所**: JSX に `{confirmDialog}` を置き忘れると**エラーも出ず静かに機能しない**（confirm の Promise が永久未解決）。UI が hook に内包され再利用性・テスト容易性が下がる。
- **重要な落とし穴（再マウント）**: hook 内で `React.FC`（コンポーネント）を定義して返すと、フック呼び出し / state 変更のたびに別の関数オブジェクトになり、React が別コンポーネントとみなして**強制アンマウント→再マウント**する（fizumi 氏の検証あり。1 秒ごとの強制再レンダーで `render` 直後に `unmount` が出る）。結果、内部 state がリセットされ `useEffect` のマウント処理が再実行され、アニメーションも効かなくなる。**回避策**: hook の内側で毎回コンポーネント関数を新規生成しないこと。`JSX.Element` を直接返す（上記）、`() => JSX.Element`（レンダー関数）を返す、または `React.FC` を hook 外で静的に定義して `[Dialog, props]` のセットで返す、のいずれかにする。

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、Context 不要 | D（Provider なし命令的、react-confirm 等） |
| List ローカルで完結、Provider を導入したくない | A or E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・stale closure・アンマウント時 pending・concurrent 相性の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。

## まとめ

1. 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
2. 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
3. Promise 方式（C/D/E）の「線形に書ける」利点は、resolver という可変外部状態を持つことの代償（4 つの罠）とセット。material-ui-confirm が v4 で「常に resolve + reason」へ倒したのは、この罠への公式の回答。
4. 「Promise の罠を避けたいが、Provider の統一感は欲しい」要件には B が最適。
5. Render Hook（E）は日本ローカルで議論されるパターン。JSX 配置忘れと「hook 内でコンポーネントを定義して返すと再マウント」の落とし穴に注意。
