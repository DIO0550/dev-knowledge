## TL;DR

- 実装方式は 3 つの軸の組み合わせ: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({onConfirm, onCancel})`）
  3. **Provider + Promise 方式**（`await confirm({...})` - material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカル）
- 「Provider 方式 = Promise ベース」とは限らない。コールバック方式は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure 等の罠を回避できる。

## このドキュメントの射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を、実装可能な完全コードで比較する。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準を示す。

## 軸の整理

混同しがちな 3 つの軸を分離する:

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の useState / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onConfirm` 等を渡す） / (β) コールバック方式（`show({onConfirm})`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称。`Provider 方式 ≠ Promise 方式` であることに注意。

## 共通の前提コード

以下、サンプル全体で使う型と削除関数:

```tsx
// types.ts
export type Item = {
  id: string;
  name: string;
};

// api.ts
export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: 'DELETE' });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

UI ライブラリは shadcn/ui の `AlertDialog` を想定（Radix UI ベース、destructive style 対応）。

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブを提供するだけで、state は呼び出し側で `useState` で管理する。

- **state**: 呼び出し側
- **API**: props 駆動
- **UI**: 利用側組み立て
- **長所**: 仕組みが透明、ローカル state と自然に連携
- **短所**: 各 List ごとに `useState` と Dialog を書く必要

```tsx
// DeleteItemButton.tsx
import { useState } from 'react';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel,
  AlertDialogContent, AlertDialogDescription, AlertDialogFooter,
  AlertDialogHeader, AlertDialogTitle, AlertDialogTrigger,
} from '@/components/ui/alert-dialog';
import { Button } from '@/components/ui/button';
import { deleteItem } from './api';
import type { Item } from './types';

export function DeleteItemButton({ item, onDeleted }: { item: Item; onDeleted?: () => void }) {
  const [open, setOpen] = useState(false);
  const [pending, setPending] = useState(false);

  const handleConfirm = async () => {
    setPending(true);
    try {
      await deleteItem(item.id);
      onDeleted?.();
      setOpen(false);
    } finally {
      setPending(false);
    }
  };

  return (
    <AlertDialog open={open} onOpenChange={setOpen}>
      <AlertDialogTrigger asChild>
        <Button variant="destructive">削除</Button>
      </AlertDialogTrigger>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>「{item.name}」を削除しますか？</AlertDialogTitle>
          <AlertDialogDescription>この操作は取り消せません。</AlertDialogDescription>
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel disabled={pending}>キャンセル</AlertDialogCancel>
          {/* preventDefault で、非同期完了まで閉じない */}
          <AlertDialogAction
            disabled={pending}
            onClick={(e) => { e.preventDefault(); handleConfirm(); }}
          >
            {pending ? '削除中...' : '削除'}
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );
}
```

## パターン B: Provider + コールバック方式

App root に `<ConfirmProvider>` を 1 つ置き、利用側は `showConfirm({onConfirm, onCancel})` を呼ぶ。**Promise の resolve トリックを使わず、コールバックで結果を受け取る**。

- **state**: Provider が保持
- **API**: コールバック
- **UI**: ライブラリ（Provider 内）が提供
- **長所**: Promise の resolve トリック不要、`onConfirm: async () => {}` がそのまま使える、アンマウント問題が起きにくい
- **短所**: 呼び出し側コードが線形にならない

`onConfirm` の戻り値 Promise を Provider が `await` する設計により、loading 状態（`pending`）を Provider 側で一元管理できる。

```tsx
// ConfirmProvider.tsx
import { createContext, useContext, useState, type ReactNode } from 'react';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ConfirmOptions = {
  title: string;
  description?: string;
  confirmText?: string;
  onConfirm: () => void | Promise<void>; // async をそのまま渡せる
  onCancel?: () => void;
};

const ConfirmContext = createContext<((o: ConfirmOptions) => void) | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [opts, setOpts] = useState<ConfirmOptions | null>(null);
  const [pending, setPending] = useState(false);

  const showConfirm = (o: ConfirmOptions) => setOpts(o);
  const close = () => { setOpts(null); setPending(false); };

  const handleConfirm = async () => {
    if (!opts) return;
    setPending(true);
    try {
      await opts.onConfirm(); // 戻り値 Promise を Provider が await → loading 一元管理
      close();
    } catch {
      setPending(false); // 失敗時はダイアログを開いたまま
    }
  };

  return (
    <ConfirmContext.Provider value={showConfirm}>
      {children}
      <AlertDialog
        open={opts !== null}
        onOpenChange={(o) => { if (!o && !pending) { opts?.onCancel?.(); close(); } }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>{opts?.title}</AlertDialogTitle>
            {opts?.description && <AlertDialogDescription>{opts.description}</AlertDialogDescription>}
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel disabled={pending}>キャンセル</AlertDialogCancel>
            <AlertDialogAction disabled={pending} onClick={(e) => { e.preventDefault(); handleConfirm(); }}>
              {pending ? '処理中...' : (opts?.confirmText ?? 'OK')}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm() {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error('useConfirm must be used within ConfirmProvider');
  return ctx;
}
```

```tsx
// 呼び出し側
import { useConfirm } from './ConfirmProvider';
import { deleteItem } from './api';
import type { Item } from './types';

export function DeleteButton({ item }: { item: Item }) {
  const showConfirm = useConfirm();
  return (
    <button onClick={() => showConfirm({
      title: `「${item.name}」を削除しますか？`,
      description: 'この操作は取り消せません。',
      confirmText: '削除',
      onConfirm: async () => { await deleteItem(item.id); }, // resolve トリック不要
    })}>削除</button>
  );
}
```

このコールバック方式の実ライブラリ例として **react-confirm-hook (DASPRiD)** がある。`ConfirmProvider` で囲み、`onConfirm` が `void | Promise<void>` を返すコールバック方式で、まさにこのパターン B に該当する。「Provider 方式は自前実装が多い」と言われがちだが、ライブラリ実例も存在する。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`await confirm({...})` で線形に書ける。

- **state**: Provider + `useRef` で resolver 保持
- **API**: Promise
- **UI**: ライブラリ（Provider 内）が提供
- **長所**: 呼び出し側コードが線形（`if (await confirm()) {...}`）、既存ライブラリと API 一致
- **短所**: resolver を `useRef` で保持する命令的仕掛け、resolve 漏れ・stale closure のリスク

material-ui-confirm v4 では、`confirm()` の戻り値が `Promise<{ confirmed: boolean; reason: 'confirm' | 'cancel' | 'natural' | 'unmount' }>` に変わった。`reason: 'unmount'` は「ダイアログの親がツリーからアンマウントされて閉じた」ケースを表し、v3 時代の「アンマウント時に Promise が永遠に pending」問題を解消している（`natural` は背景クリック等の自然な閉じ）。v3 互換挙動が必要なら `useLegacyReturn` プロップで復元できる。

```tsx
// ConfirmProvider.tsx
import { createContext, useContext, useEffect, useRef, useState, type ReactNode } from 'react';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ConfirmOptions = { title: string; description?: string; confirmText?: string };
type ConfirmResult = { confirmed: boolean; reason: 'confirm' | 'cancel' | 'unmount' };
type ConfirmFn = (o: ConfirmOptions) => Promise<ConfirmResult>;

const ConfirmContext = createContext<ConfirmFn | null>(null);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [opts, setOpts] = useState<ConfirmOptions | null>(null);
  const resolverRef = useRef<((r: ConfirmResult) => void) | null>(null);

  const confirm: ConfirmFn = (o) =>
    new Promise<ConfirmResult>((resolve) => {
      resolverRef.current = resolve;
      setOpts(o);
    });

  const settle = (r: ConfirmResult) => {
    resolverRef.current?.(r);      // resolve 漏れ防止: 必ず resolver 経由で 1 回 resolve
    resolverRef.current = null;
    setOpts(null);
  };

  // アンマウント時に pending を残さない（material-ui-confirm v4 の reason:'unmount' 相当）
  useEffect(() => () => resolverRef.current?.({ confirmed: false, reason: 'unmount' }), []);

  return (
    <ConfirmContext.Provider value={confirm}>
      {children}
      <AlertDialog
        open={opts !== null}
        onOpenChange={(o) => { if (!o) settle({ confirmed: false, reason: 'cancel' }); }}
      >
        <AlertDialogContent>
          <AlertDialogHeader>
            <AlertDialogTitle>{opts?.title}</AlertDialogTitle>
            {opts?.description && <AlertDialogDescription>{opts.description}</AlertDialogDescription>}
          </AlertDialogHeader>
          <AlertDialogFooter>
            <AlertDialogCancel onClick={() => settle({ confirmed: false, reason: 'cancel' })}>キャンセル</AlertDialogCancel>
            <AlertDialogAction onClick={() => settle({ confirmed: true, reason: 'confirm' })}>{opts?.confirmText ?? 'OK'}</AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export function useConfirm() {
  const ctx = useContext(ConfirmContext);
  if (!ctx) throw new Error('useConfirm must be used within ConfirmProvider');
  return ctx;
}
```

```tsx
// 呼び出し側 — 線形に書ける
const confirm = useConfirm();
const onDelete = async (item: Item) => {
  const { confirmed } = await confirm({ title: `「${item.name}」を削除しますか？`, confirmText: '削除' });
  if (confirmed) await deleteItem(item.id);
};
```

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、グローバル関数として `import { confirm } from './confirm'`。内部で React 18 の `createRoot` で Dialog を `document.body` に append した `div` を root にして動的マウントする。

- **state**: React tree 外（別 root が管理）
- **API**: Promise
- **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む
- **長所**: Provider セットアップ不要、`window.confirm()` のメンタルモデル
- **短所**: デフォルトでは Context にアクセス不可（別 root のため）。`ContextAwareConfirmation.ConfirmationRoot` を root に置けば対応可（実質ミニ Provider）

```tsx
// confirm.tsx
import { confirmable, createConfirmation, type ConfirmDialogProps } from 'react-confirm';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type Props = { title: string; description?: string; confirmText?: string };

// confirmable が show/proceed/cancel を注入。proceed(true)/proceed(false) で Promise を解決
const ConfirmDialog = ({ show, proceed, title, description, confirmText }: ConfirmDialogProps<Props, boolean>) => (
  <AlertDialog open={show} onOpenChange={(o) => { if (!o) proceed(false); }}>
    <AlertDialogContent>
      <AlertDialogHeader>
        <AlertDialogTitle>{title}</AlertDialogTitle>
        {description && <AlertDialogDescription>{description}</AlertDialogDescription>}
      </AlertDialogHeader>
      <AlertDialogFooter>
        <AlertDialogCancel onClick={() => proceed(false)}>キャンセル</AlertDialogCancel>
        <AlertDialogAction onClick={() => proceed(true)}>{confirmText ?? 'OK'}</AlertDialogAction>
      </AlertDialogFooter>
    </AlertDialogContent>
  </AlertDialog>
);

// createConfirmation がデフォルト mounter（createRoot で document.body にマウント）を使う
export const confirm = createConfirmation(confirmable(ConfirmDialog));
```

```tsx
// 呼び出し側 — Provider 不要、グローバル関数
import { confirm } from './confirm';
import { deleteItem } from './api';
import type { Item } from './types';

export async function onDelete(item: Item) {
  const ok = await confirm({ title: `「${item.name}」を削除しますか？`, confirmText: '削除' });
  if (ok) await deleteItem(item.id);
}
```

テーマ等の Context が必要なら `ContextAwareConfirmation.createConfirmation` を使い、アプリツリーに `ContextAwareConfirmation.ConfirmationRoot` を置く（こちらは `createPortal` でツリー内に保つ別経路）。

## パターン E: Render Hook（hook が JSX を返す）

LINE 証券のテックブログが初出、uhyo 氏が「render hooks」として記事化（記事は uhyo 氏本人が執筆、命名は uhyo 氏の当時の上司）。日本ローカルで議論されるパターン。

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録）
- **API**: コールバック + JSX 返却
- **UI**: hook 内で JSX 構築
- **長所**: Provider 不要、ローカルに閉じる
- **短所**: JSX に `{confirmDialog}` を置き忘れると静かに動かない、UI が hook に内包され再利用性低下、命名者自身が「流行る気配がない」と評している、英語圏ライブラリ採用例なし

**重要な落とし穴**: hook 内で `React.FC` を定義して返すと、Custom Hook 呼び出し毎に再マウントされる（fneco 氏の検証あり）。必ず `JSX.Element` を直接返すか `() => JSX.Element` を返す形式にする（別記事「render hooks パターン」参照）。

```tsx
// useDeleteConfirm.tsx
import { useCallback, useRef, useState } from 'react';
import {
  AlertDialog, AlertDialogAction, AlertDialogCancel, AlertDialogContent,
  AlertDialogDescription, AlertDialogFooter, AlertDialogHeader, AlertDialogTitle,
} from '@/components/ui/alert-dialog';

export function useDeleteConfirm() {
  const [open, setOpen] = useState(false);
  const [target, setTarget] = useState<{ name: string } | null>(null);
  const onConfirmRef = useRef<(() => void | Promise<void>) | null>(null);

  // hooks を呼ばない。state からの JSX.Element を返すだけ（再マウントしない本来の形）
  const confirmDialog = (
    <AlertDialog open={open} onOpenChange={setOpen}>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>「{target?.name}」を削除しますか？</AlertDialogTitle>
          <AlertDialogDescription>この操作は取り消せません。</AlertDialogDescription>
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel>キャンセル</AlertDialogCancel>
          <AlertDialogAction
            onClick={async () => { await onConfirmRef.current?.(); setOpen(false); }}
          >削除</AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );

  const confirmDelete = useCallback((item: { name: string }, onConfirm: () => void | Promise<void>) => {
    setTarget(item);
    onConfirmRef.current = onConfirm;
    setOpen(true);
  }, []);

  return { confirmDialog, confirmDelete }; // JSX.Element を返す（FC は返さない）
}
```

```tsx
// 呼び出し側 — JSX に {confirmDialog} を置き忘れると静かに動かない点が短所
import { useDeleteConfirm } from './useDeleteConfirm';
import { deleteItem } from './api';
import type { Item } from './types';

export function ItemList({ items }: { items: Item[] }) {
  const { confirmDialog, confirmDelete } = useDeleteConfirm();
  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>
          {item.name}
          <button onClick={() => confirmDelete(item, () => deleteItem(item.id))}>削除</button>
        </li>
      ))}
      {confirmDialog /* ← 置き忘れ注意 */}
    </ul>
  );
}
```

## 全パターン比較表

| | A: プリミティブ | B: Provider + callback | C: Provider + Promise | D: Provider なし命令的 | E: Render Hook |
|---|---|---|---|---|---|
| **state の置き場** | 呼び出し側 | Provider | Provider + useRef | React tree 外 | hook 内（実体は呼び出し元） |
| **API スタイル** | props 駆動 | コールバック | Promise（await） | Promise（await） | コールバック + JSX |
| **Provider 必要** | 不要 | 必要 | 必要 | 不要 | 不要 |
| **Context アクセス** | 自然 | 自然 | 自然 | デフォルト不可 | 自然 |
| **resolve 漏れリスク** | なし | なし | あり（要対策） | ライブラリが対策 | なし |
| **await 跨ぎ stale closure** | なし | なし | あり（要規律） | あり（要規律） | なし |
| **呼び出し側コードの線形性** | △ | △ | ◎ | ◎ | △ |
| **再利用範囲** | 各 List で重複 | アプリ全体 | アプリ全体 | アプリ全体 | hook 再利用可 |
| **ライブラリ採用例** | shadcn / Radix | react-confirm-hook 等 | material-ui-confirm 等多数 | react-confirm 等 | 確認できず |

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、Context 不要 | D（Provider なし命令的、react-confirm 等） |
| List ローカルで完結、Provider を導入したくない | A or E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・stale closure 等の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。

## まとめ

- 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
- 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
- 「Promise の罠を避けたいが、Provider の統一感は欲しい」要件には B が最適。
- ライブラリ採用例の多さは `C > D > A` の順。E（Render Hook）はライブラリ採用例なし。

## 環境・前提

- React 18 系（パターン D の `react-confirm` は `react-dom/client` の `createRoot` を使うため React 18+ が前提。react-confirm 0.2.x / 0.3.x）。
- material-ui-confirm は v4 を前提（`{ confirmed, reason }` API）。
- UI プリミティブは shadcn/ui の `AlertDialog`（Radix UI ベース）。
