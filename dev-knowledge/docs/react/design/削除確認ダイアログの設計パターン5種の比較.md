## TL;DR

- 実装方式は 3 つの軸の組み合わせ: **state の置き場**（呼び出し側 / Provider / React tree 外）、**呼び出し API**（props / コールバック / Promise）、**UI の提供形式**（ライブラリ提供 / 利用側組み立て）。
- 実在する組み合わせは大きく 5 パターン:
  1. **プリミティブ + 呼び出し側 state**（shadcn / Radix）
  2. **Provider + コールバック方式**（`showConfirm({ onConfirm, onCancel })`）
  3. **Provider + Promise 方式**（`await confirm({...})` — material-ui-confirm 系）
  4. **Provider なし命令的 API**（react-confirm 系、`createRoot` で動的マウント）
  5. **Render Hook**（hook が JSX を返す。日本ローカル）
- 「Provider 方式 = Promise ベース」とは限らない。コールバック方式は Promise の resolve トリックを使わずに済むため、resolve 漏れ・stale closure 等の罠を回避できる。

## このドキュメントの射程

- React で「アイテム削除時に確認ダイアログを出す」UI の実装方式を、実装可能なコードで比較する。
- 5 パターンそれぞれの長所・短所・トレードオフを示す。
- 削除以外でも応用できる「確認ダイアログ」設計の決定基準。

## 軸の整理

混同しがちな 3 つの軸を分離する:

| 軸 | 選択肢 |
|---|---|
| state の置き場 | (a) 呼び出し側の useState / (b) Provider の Context / (c) React tree 外（`createRoot` で動的マウント） |
| 呼び出し API | (α) props 駆動（`open`, `onConfirm` 等を渡す） / (β) コールバック方式（`show({ onConfirm })`） / (γ) Promise 方式（`await confirm()`） |
| Dialog UI | (i) ライブラリが提供 / (ii) 利用側が組み立て（プリミティブ） |

「Promise-based」「Provider パターン」「Render Hook」はこの 3 軸の組み合わせの呼称。`Provider 方式 ≠ Promise 方式` であることに注意。

## 共通の前提コード

```ts
// types.ts
export type Item = {
  id: string;
  name: string;
};

// api.ts
export async function deleteItem(id: string): Promise<void> {
  const res = await fetch(`/api/items/${id}`, { method: 'DELETE' });
  if (!res.ok) throw new Error(`Failed to delete item ${id}`);
}
```

UI ライブラリは shadcn/ui の `AlertDialog`（Radix UI ベース、destructive style 対応）を想定。

---

## パターン A: プリミティブ + 呼び出し側 state（shadcn / Radix）

ライブラリは accessible なプリミティブを提供するだけで、state は呼び出し側で `useState` で管理する。

- **state**: 呼び出し側
- **API**: props 駆動
- **UI**: 利用側組み立て

```tsx
import { useState } from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';
import { deleteItem } from './api';
import type { Item } from './types';

export function ItemRow({ item }: { item: Item }) {
  const [open, setOpen] = useState(false);
  const [pending, setPending] = useState(false);

  const handleConfirm = async () => {
    setPending(true);
    try {
      await deleteItem(item.id);
      setOpen(false);
    } finally {
      setPending(false);
    }
  };

  return (
    <>
      <span>{item.name}</span>
      <button onClick={() => setOpen(true)}>削除</button>

      <AlertDialog open={open} onOpenChange={setOpen}>
        <AlertDialogContent>
          <AlertDialogTitle>削除確認</AlertDialogTitle>
          <AlertDialogDescription>
            「{item.name}」を削除します。この操作は取り消せません。
          </AlertDialogDescription>
          <AlertDialogFooter>
            <AlertDialogCancel disabled={pending}>キャンセル</AlertDialogCancel>
            <AlertDialogAction onClick={handleConfirm} disabled={pending}>
              {pending ? '削除中...' : '削除'}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </>
  );
}
```

- **長所**: 仕組みが透明、ローカル state と自然に連携。
- **短所**: 各 List ごとに `useState` と Dialog を書く必要がある。

## パターン B: Provider + コールバック方式

App root に `<ConfirmProvider>` を 1 つ置き、利用側は `showConfirm({ onConfirm, onCancel })` を呼ぶ。**Promise の resolve トリックを使わず、コールバックで結果を受け取る**。

- **state**: Provider が保持
- **API**: コールバック
- **UI**: ライブラリ（Provider 内）が提供

```tsx
// confirm-provider.tsx
import { createContext, useCallback, useContext, useState, type ReactNode } from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type ConfirmOptions = {
  title: string;
  description?: string;
  onConfirm: () => void | Promise<void>;
  onCancel?: () => void;
};

const ConfirmContext = createContext<(opts: ConfirmOptions) => void>(() => {});

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [opts, setOpts] = useState<ConfirmOptions | null>(null);
  const [pending, setPending] = useState(false);

  const showConfirm = useCallback((next: ConfirmOptions) => setOpts(next), []);

  const handleConfirm = async () => {
    if (!opts) return;
    setPending(true);
    try {
      // onConfirm の戻り値 Promise を Provider が await することで loading を一元管理
      await opts.onConfirm();
      setOpts(null);
    } finally {
      setPending(false);
    }
  };

  const handleCancel = () => {
    opts?.onCancel?.();
    setOpts(null);
  };

  return (
    <ConfirmContext.Provider value={showConfirm}>
      {children}
      <AlertDialog open={opts !== null} onOpenChange={(o) => !o && handleCancel()}>
        <AlertDialogContent>
          <AlertDialogTitle>{opts?.title}</AlertDialogTitle>
          {opts?.description && (
            <AlertDialogDescription>{opts.description}</AlertDialogDescription>
          )}
          <AlertDialogFooter>
            <AlertDialogCancel disabled={pending}>キャンセル</AlertDialogCancel>
            <AlertDialogAction onClick={handleConfirm} disabled={pending}>
              {pending ? '処理中...' : 'OK'}
            </AlertDialogAction>
          </AlertDialogFooter>
        </AlertDialogContent>
      </AlertDialog>
    </ConfirmContext.Provider>
  );
}

export const useConfirm = () => useContext(ConfirmContext);
```

```tsx
// 使う側
import { useConfirm } from './confirm-provider';
import { deleteItem } from './api';
import type { Item } from './types';

export function ItemRow({ item }: { item: Item }) {
  const showConfirm = useConfirm();

  const handleDelete = () => {
    showConfirm({
      title: '削除確認',
      description: `「${item.name}」を削除します。この操作は取り消せません。`,
      onConfirm: async () => {
        await deleteItem(item.id); // async がそのまま使える
      },
    });
  };

  return (
    <>
      <span>{item.name}</span>
      <button onClick={handleDelete}>削除</button>
    </>
  );
}
```

- **長所**: Promise の resolve トリック不要、`onConfirm: async () => {}` がそのまま使える、アンマウント問題が起きにくい。`onConfirm` の戻り値 Promise を Provider が `await` する設計により loading 状態を Provider 側で一元管理できる。
- **短所**: 呼び出し側コードが線形にならない（コールバックにロジックを渡す形）。

## パターン C: Provider + Promise 方式（material-ui-confirm 系）

ライブラリ採用例が最も多い形。`await confirm({...})` で線形に書ける。

- **state**: Provider + `useRef` で resolver 保持
- **API**: Promise
- **UI**: ライブラリ（Provider 内）が提供

```tsx
// confirm-provider.tsx（Promise 版・要点）
import { createContext, useCallback, useContext, useRef, useState, type ReactNode } from 'react';

type ConfirmOptions = { title: string; description?: string };
type Resolver = (confirmed: boolean) => void;

const ConfirmContext = createContext<(opts: ConfirmOptions) => Promise<boolean>>(
  async () => false,
);

export function ConfirmProvider({ children }: { children: ReactNode }) {
  const [opts, setOpts] = useState<ConfirmOptions | null>(null);
  const resolverRef = useRef<Resolver | null>(null);

  const confirm = useCallback((next: ConfirmOptions) => {
    setOpts(next);
    return new Promise<boolean>((resolve) => {
      resolverRef.current = resolve; // resolver を ref に保持する命令的仕掛け
    });
  }, []);

  const settle = (confirmed: boolean) => {
    resolverRef.current?.(confirmed);
    resolverRef.current = null;
    setOpts(null);
  };

  return (
    <ConfirmContext.Provider value={confirm}>
      {children}
      {/* opts を使った AlertDialog。OK で settle(true)、Cancel/クローズで settle(false) */}
    </ConfirmContext.Provider>
  );
}

export const useConfirm = () => useContext(ConfirmContext);
```

```tsx
// 使う側
const confirm = useConfirm();

const handleDelete = async () => {
  if (await confirm({ title: '削除確認', description: '取り消せません' })) {
    await deleteItem(item.id);
  }
};
```

- **長所**: 呼び出し側コードが線形（`if (await confirm()) {...}`）、既存ライブラリと API 一致。
- **短所**: resolver を `useRef` で保持する命令的仕掛け。resolve 漏れ・stale closure のリスク。

material-ui-confirm v4 では「アンマウント時に Promise が resolve されず pending のままになる」問題（v3 の挙動）を、`{ confirmed, reason }` を返す API で解決している。`reason` は `"confirm" | "cancel" | "natural" | "unmount"` の 4 値で、アンマウント時は `reason: "unmount"` として常に resolve される（旧挙動は `useLegacyReturn` で復元可能）。

## パターン D: Provider なし命令的 API（react-confirm 系）

Provider を一切置かず、グローバル関数として `import { confirm } from './confirm'`。内部で React 18 の `createRoot` で Dialog を `document.body` 直下に動的マウントする。

- **state**: React tree 外（別 root が管理）
- **API**: Promise
- **UI**: 利用側が組み立てた Dialog を `confirmable()` で包む

```tsx
// ConfirmDialog.tsx
import { confirmable, createConfirmation, type ConfirmDialogProps } from 'react-confirm';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type Props = { title: string; description?: string };

const ConfirmDialog = ({ show, proceed, title, description }: ConfirmDialogProps<Props, boolean>) => (
  <AlertDialog open={show} onOpenChange={(o) => !o && proceed(false)}>
    <AlertDialogContent>
      <AlertDialogTitle>{title}</AlertDialogTitle>
      {description && <AlertDialogDescription>{description}</AlertDialogDescription>}
      <AlertDialogCancel onClick={() => proceed(false)}>キャンセル</AlertDialogCancel>
      <AlertDialogAction onClick={() => proceed(true)}>削除</AlertDialogAction>
    </AlertDialogContent>
  </AlertDialog>
);

export const confirm = createConfirmation(confirmable(ConfirmDialog));
```

```tsx
// 使う側（Provider 不要）
import { confirm } from './ConfirmDialog';

const handleDelete = async () => {
  if (await confirm({ title: '削除確認', description: '取り消せません' })) {
    await deleteItem(item.id);
  }
};
```

- **長所**: Provider セットアップ不要、`window.confirm()` のメンタルモデル。
- **短所**: デフォルトでは Context にアクセスできない（別 root のため）。`ContextAwareConfirmation.ConfirmationRoot` を Provider 配下の root に置けば対応可能（実質ミニ Provider で、内部の `reactTree` マウンターを使う）。

## パターン E: Render Hook（hook が JSX を返す）

LINE 証券のテックブログが初出、uhyo 氏が「render hooks」として記事化（命名は uhyo 氏の当時の上司）。日本ローカルで議論されるパターン。

- **state**: hook 内（実体は呼び出し元コンポーネントのフックとして登録）
- **API**: コールバック + JSX 返却
- **UI**: hook 内で JSX 構築

```tsx
// useConfirm.tsx
import { useCallback, useState } from 'react';
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogTitle,
} from '@/components/ui/alert-dialog';

type Options = { title: string; description?: string; onConfirm: () => void | Promise<void> };

export function useConfirm() {
  const [opts, setOpts] = useState<Options | null>(null);

  const confirm = useCallback((next: Options) => setOpts(next), []);

  // ✅ JSX.Element を直接返す（FC を hook 内で定義しない）
  const confirmDialog = (
    <AlertDialog open={opts !== null} onOpenChange={(o) => !o && setOpts(null)}>
      <AlertDialogContent>
        <AlertDialogTitle>{opts?.title}</AlertDialogTitle>
        {opts?.description && <AlertDialogDescription>{opts.description}</AlertDialogDescription>}
        <AlertDialogCancel>キャンセル</AlertDialogCancel>
        <AlertDialogAction
          onClick={async () => {
            await opts?.onConfirm();
            setOpts(null);
          }}
        >
          削除
        </AlertDialogAction>
      </AlertDialogContent>
    </AlertDialog>
  );

  return { confirm, confirmDialog };
}
```

```tsx
// 使う側
export function ItemRow({ item }: { item: Item }) {
  const { confirm, confirmDialog } = useConfirm();
  return (
    <>
      <button
        onClick={() =>
          confirm({ title: '削除確認', onConfirm: () => deleteItem(item.id) })
        }
      >
        削除
      </button>
      {confirmDialog /* ← JSX への配置を忘れると静かに動かない */}
    </>
  );
}
```

- **長所**: Provider 不要、ローカルに閉じる。
- **短所**: JSX に `{confirmDialog}` を置き忘れると静かに動かない、UI が hook に内包され再利用性低下、命名者自身が「流行る気配がない」と評している、（主要ライブラリでの）採用例は確認できず。
- **重要な落とし穴**: hook 内で `React.FC` を定義して返すと、Custom Hook が再実行されるたびに再マウントされる（別記事「render hooks パターン」参照）。必ず `JSX.Element` を直接返すか `() => JSX.Element` を返す形式にする。

## 全パターン比較表

| | A: プリミティブ | B: Provider + callback | C: Provider + Promise | D: Provider なし命令的 | E: Render Hook |
|---|---|---|---|---|---|
| **state の置き場** | 呼び出し側 | Provider | Provider + useRef | React tree 外 | hook 内（実体は呼び出し元） |
| **API スタイル** | props 駆動 | コールバック | Promise（await） | Promise（await） | コールバック + JSX |
| **Provider 必要** | 不要 | 必要 | 必要 | 不要 | 不要 |
| **Context アクセス** | 自然 | 自然 | 自然 | デフォルト不可 | 自然 |
| **resolve 漏れリスク** | なし | なし | あり（要対策） | ライブラリが対策 | なし |
| **await 跨ぎ stale closure** | なし | なし | あり（要規律） | あり（要規律） | なし |
| **呼び出し側コードの線形性** | △ | △ | ◎ | ◎ | △ |
| **再利用範囲** | 各 List で重複 | アプリ全体 | アプリ全体 | アプリ全体 | hook 再利用可 |
| **ライブラリ採用例** | shadcn / Radix | 自前実装が多い（react-confirm-hook 等） | material-ui-confirm 等多数 | react-confirm 等 | 確認できず |

## 選択基準

| 規模・要件 | 推奨 |
|---|---|
| 1〜2 箇所だけ、自社の Dialog コンポーネントがある | A（プリミティブ + 呼び出し側 state） |
| 中規模・複数箇所、Promise の罠を避けたい | **B（Provider + コールバック）** |
| 中規模・複数箇所、`await confirm()` で線形に書きたい | C（Provider + Promise） |
| Provider セットアップを避けたい、Context 不要 | D（Provider なし命令的、react-confirm 等） |
| List ローカルで完結、Provider を導入したくない | A or E（Render Hook） |
| 削除以外も含めた汎用確認 UI を全社で揃えたい | B or C |

「Provider + Promise」は呼び出し側の書き味は良いが、resolve 漏れ・stale closure 等の罠を引き受ける必要がある。**Promise の罠を避けたいなら B（コールバック方式）が筋がいい**。

## まとめ

- 「Provider 方式 = Promise ベース」ではない。Provider + コールバック方式（パターン B）も実用的な選択肢。
- 軸を分解すると 5 パターン: プリミティブ / Provider+callback / Provider+Promise / Provider なし命令的 / Render Hook。
- 「Promise の罠を避けたいが、Provider の統一感は欲しい」要件には B が最適。
- ライブラリ採用例の多さは `C > D > A` の順。E（Render Hook）は主要ライブラリでの採用例なし。
