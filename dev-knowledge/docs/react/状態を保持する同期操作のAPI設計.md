## このドキュメントの射程

「ミュータブルな状態を保持するオブジェクトに対し、状態を変える同期操作を行う。
その操作が**失敗しうる**(不変条件で弾く)とき、結果をどう呼び出し側に伝えるか」
という設計問題を扱う。

これは特定のフレームワークの問題ではない。以下はすべて同じ骨格の問題:

- Rust で `Vec` のラッパー型に `try_push` を生やす — `Result` を返すか、
  エラーを内部に溜めるか
- DDD の集約メソッドが不変条件違反をどう表現するか — 例外 / Result型 /
  ドメインイベント
- React のカスタムフックで配列に `add` する — 戻り値 / コールバック /
  保持された state

本文では一例として配列(コレクション)への要素追加を使うが、論点はコレクションに
限らない。Reactを使う場合は最後の「環境固有の制約」セクションを参照。

## 中心となる設計原則

### CQS(Command Query Separation)

- **コマンド**: 状態を変える。値を返さない(`void`)。
- **クエリ**: 値を返す。状態を変えない(副作用なし)。
- 1つの関数が両方をやると責務が混ざる。

重要な注意: CQSは「**1つの関数が状態変更と値返却の両方をやるな**」であって、
「結果を返す関数を持つな」ではない。結果を返す責務を別の関数(クエリ)や
別の経路(コールバック・保持された状態)に切り出せば、原則は守れる。

### ファーストクラスコレクション / カプセル化

不変条件(何を入れてよいか)は、それを保持するオブジェクト自身が守るべき。
判定ロジックを外に出して呼び出し側に `if (canAdd()) add()` させると、
ルールがオブジェクトの外に散らばる。「弾く」はコレクションが持つ不変条件であり、
内部に閉じているのが望ましい。

### 「破壊的操作の戻り値で結果を返す」ことへの違和感

命令的な世界では、破壊的変更の副産物として状態を返すAPIが一般的
(例: `Array.push` は新しい `length` を返す)。
だがこれは CQS 違反であり、結果の在り処を二重化させる。
特にイミュータブル・宣言的な環境では、状態は「次に観測される状態」に現れるのが
本筋なので、戻り値経由の結果伝達は真実の源(source of truth)を二重化させて
混乱を招く。

## 結果の伝え方 — 3つのパターン

失敗しうる状態変更操作で、結果(成功/失敗、失敗の理由)をどう外に出すか。
違いは **「結果がどこに、いつ現れるか」**。

### パターン1: 結果自体を状態として保持する

操作対象オブジェクトが、成功分と並んで**失敗分も内部状態として蓄積**する。
結果が state に残るのが特徴。一過性の通知では消える情報を保持し続けられる。

```ts
type RejectReason = "DUPLICATE" | "LIMIT_EXCEEDED" | "INVALID";

// 例: コレクション。実体はクラスでもフックでも構造体でもよい
class Collection<T> {
  private items: T[] = [];
  private rejections: { item: T; reason: RejectReason }[] = [];

  add(item: T): void {              // コマンド: void に徹する
    const reason = this.findReason(item);
    if (reason) {
      this.rejections.push({ item, reason });  // 失敗も内部に保持
      return;
    }
    this.items.push(item);
  }

  getItems() { return this.items; }
  getRejections() { return this.rejections; }  // クエリで観測
}
```

- **結果の置き場所**: 状態(永続)
- **観測タイミング**: 操作後、状態を読みにいく
- **向く用途**: 弾かれた一覧を残したい/後で参照したい
- **トレードオフ**: 状態が増える。いつクリアするか(寿命管理)を考える必要

### パターン2: コールバック(継続)を受け取って結果を流す

操作に継続(continuation)を渡し、結果をそこへ流す。
「コマンドが結果を返す」のではなく「オブジェクトが起きたことを通知する」構図。
状態には残さず、その場で発火するだけ。

```ts
type AddOutcome<T> =
  | { type: "ADDED"; item: T }
  | { type: "REJECTED"; item: T; reason: RejectReason };

add(item: T, onResult?: (outcome: AddOutcome<T>) => void): void {
  const reason = this.findReason(item);
  if (reason) {
    onResult?.({ type: "REJECTED", item, reason });
    return;
  }
  this.items.push(item);
  onResult?.({ type: "ADDED", item });
}
```

- **結果の置き場所**: イベント(一過性)
- **観測タイミング**: 操作時に発火
- **向く用途**: トースト・ログ等「一度反応すれば終わり」の即時反応
- **トレードオフ**: 結果が残らない。判定ロジックは内部に閉じる
  (ファーストクラスコレクションと好相性)

### パターン3: チェックと操作を分離する(述語クエリ + コマンド)

判定を副作用のないクエリ(`canAdd` / `validate`)として切り出し、
コマンド(`add`)と分ける。結果は**操作の前に問い合わせる**。CQS的に最もクリーン。

```ts
// クエリ: 結果を返すが副作用なし
validate(item: T): RejectReason | null {
  return this.findReason(item);
}
// コマンド: void。内部でも必ず防御(無条件 add を防ぐ)
add(item: T): void {
  if (this.findReason(item)) return;   // すり抜け防止の最終ガード
  this.items.push(item);
}
```

```ts
// 呼び出し側が順序を制御
const reason = c.validate(item);
if (reason) showMessage(reason);
else c.add(item);
```

- **結果の置き場所**: クエリの戻り値
- **観測タイミング**: 操作の前に問い合わせ
- **向く用途**: 事前バリデーション、UIの事前無効化(ボタン disabled 等)
- **トレードオフ**: 不変条件の判断が「外に晒される」。
  カプセル化(ファーストクラスコレクション)とは相性が悪い。
  判定→操作の間に状態が変わる TOCTOU の理論的余地(同一同期スコープ内なら実害なし)

## 比較

| | 結果の置き場所 | タイミング | 向く用途 | カプセル化 |
|---|---|---|---|---|
| 1. 結果を保持 | 状態(永続) | 操作後に観測 | 弾かれた一覧をUIに残す | ○ |
| 2. コールバック | イベント(一過性) | 操作時に発火 | トースト・ログ等の即時反応 | ◎ |
| 3. チェック分離 | クエリの戻り値 | 操作前に問い合わせ | 事前無効化・バリデーション | △(外に晒す) |

**3つは排他ではなく組み合わせられる。** 例: 3で事前に止めつつ、
すり抜けた分は2で通知し、1で履歴も残す。
選択軸は (a) 結果の寿命(一過性か永続か)、(b) 判定が操作の前に欲しいか後でいいか、
(c) カプセル化をどこまで重視するか。

## 共通: 判定ロジックの一元化

どのパターンでも、判定が複数箇所に重複しがち(`validate` と `add` 内など)。
ロジックを1関数にまとめ、適用する対象だけ変える。

```ts
private findReason(item: T): RejectReason | null {
  if (this.items.some((i) => this.keyOf(i) === this.keyOf(item))) return "DUPLICATE";
  if (this.max !== undefined && this.items.length >= this.max) return "LIMIT_EXCEEDED";
  return null;
}
```

## 失敗理由の構造化(型の指針)

失敗理由は、機械可読のコードと表示用メッセージを分離すると実務的に強い。
`react-dropzone` の `FileRejection` がよい手本:

```ts
interface FileError {
  message: string;  // 表示用
  code: string;     // 機械可読(例: "file-too-large")
}
interface FileRejection {
  file: File;
  errors: FileError[];
}
```

`{ type: "REJECTED"; item; reason }` のような判別可能ユニオンも同系統。
`code`(分岐用)と `message`(表示用)を分けておくと、UIとロジックが疎結合になる。

## 落とし穴: 状態の二重管理

「操作対象オブジェクト内部の状態」と「呼び出し側が別に持つ状態」が
二重化すると、両者がズレる。

実例(react-dropzone): 同じファイルを削除→再追加しようとすると、
ライブラリ内部 state にまだ残っていて受理されず、拒否コールバックも呼ばれないため
再追加をハンドリングできない、という報告がある。これは内部状態と外部状態の
二重管理から来るズレ。

対策: 状態は1箇所に寄せる。やむを得ず分ける場合、判定は常に一方を基準に行い、
同期を明示的に保つ。

## 言語・環境ごとの現れ方

同じ骨格が、各環境の慣習に応じて違う顔をする。

- **Rust**: `try_push(item) -> Result<(), AddError>` は戻り値で結果を返すが、
  Rust では `Result` を返すのが自然で CQS 違反の違和感は薄い
  (例外のない言語で結果型が一級市民のため)。
  内部に溜める設計(パターン1)や、エラー型に理由を持たせる設計も取れる。
- **DDD の集約**: 不変条件違反を例外で表す / `Result` で返す /
  ドメインイベントとして発行する(パターン2に近い)。
  集約はファーストクラスコレクションそのものなので、ルールは内部に閉じる。
- **汎用ユーティリティ**: 「何を入れてよいか」はドメイン固有なので、
  汎用ライブラリは判定を持たず素朴な操作だけ提供し、不変条件は利用側に委ねる
  のが通例(後述)。

## 補足: React 環境固有の制約

例として React のカスタムフックで実装する場合、上記の一般論に加えて
React 特有の制約が**上乗せ**される。これらは CQS やカプセル化そのものではなく、
「純粋関数で状態遷移を表す」という React の選択がもたらすローカルなルール。

- **真実の源は次レンダーの state**: 操作の結果は基本、再レンダー後の state に
  現れる。だから「戻り値で結果」はとりわけ噛み合わない(パターン3の事前クエリか、
  パターン1の保持 state、パターン2の通知に倒すのが自然)。
- **setState の updater は純粋に保つ**: updater 内で副作用(コールバック発火・
  外部変数代入)を行わない。
  - StrictMode の開発時二重実行で updater が複数回呼ばれる
  - バッチング下では `setState` 直後に updater が走る保証がなく、
    updater 経由で結果を取り出すと未確定値を掴む
  → **判定とコールバック発火は updater の「外」で行う。**
- **汎用リストフックは判定を持たない**: `react-use` / `@uidotdev/usehooks` /
  `reactuse` などの `useList` 系は `push` が `void` で判定も通知もない。
  不変条件はドメイン固有なので、汎用層は素朴な操作だけ提供し、判定は
  自作フック(=ドメイン層)に委ねる役割分担。「汎用 add は判定しない」は
  この層分離の結果であって、矛盾ではない。
- **コールバック方式の手本**: `react-dropzone` の
  `onDrop(accepted, rejections)` / `onDropAccepted` / `onDropRejected` は
  パターン2の代表例。かつ `acceptedFiles` / `fileRejections` を state として
  保持(パターン1)も併用しており、1と2が併存できる実例。

## 結論

- 失敗しうる状態変更操作の結果は、「戻り値で返す」より
  **保持(1) / 通知(2) / 事前クエリ(3)** のいずれか、または組み合わせで伝える。
- 破壊的操作の戻り値で結果を返すのは CQS 違反かつ真実の源の二重化を招くため、
  特に宣言的環境では避ける。ただし Rust の `Result` のように、環境の慣習次第で
  違和感の度合いは変わる。
- どれを選ぶかは「結果の寿命」「判定が操作の前に欲しいか」「カプセル化の重視度」で
  決まる。カプセル化を最優先するならコールバック通知(2)、CQSの明快さと事前UI制御
  を取るならチェック分離(3)、履歴を残すなら保持(1)。
- 判定ロジックは1箇所に一元化し、失敗理由は code/message を分離して構造化する。
- 状態の二重管理は最大の落とし穴。状態は1箇所に寄せる。
- React 等のフレームワークは、この一般問題に環境固有の制約(純粋 updater 等)を
  上乗せしているだけ。土台の設計問題を押さえれば、環境が変わっても応用が効く。
