---
title: リファクタリングは「逸脱検出」— 全面書き直しより既存の良い構造からの逸脱を潰す
tags: [rust, architecture, refactoring, domain-driven-design, smart-constructor]
---

## 問題

バックエンドを「ドメイン中心 + impl ベース」の理想形に書き直す計画を立てた。当初は全面書き直しを想定していた。

## 原因

全面書き直しは、既にある良い構造まで壊してやり直すことになり、コストとリスクが大きい。実際に現状を調査してみると、理想形への**逸脱は 5 箇所だけ**だった。

検出した逸脱:

1. `Config` に smart constructor がない（不変条件を型で守れていない）
2. `Intent` が raw なまま（値オブジェクト化されていない）
3. VO（Value Object）がない
4. 一部で `std::fs::write` を直呼びしている（副作用が境界に閉じていない）
5. 未配線の command がある

つまり土台は既に理想に近く、直すべきは「理想からズレている部分」だけだった。

## 解決

**全面書き直しをやめ、逸脱箇所を細粒度で潰す方針に転換**した。

- 作業を Epic + サブ Issue 13 件に分解し、逸脱ごとに小さな単位で修正できるようにした。
- 各サブ Issue は「どの逸脱を、どの理想形に寄せるか」が明確な単位にする（例: 「Config に smart constructor を追加」「Intent を VO 化」）。

```text
Epic: BE をドメイン中心 + impl ベースへ
├─ #xx Config に smart constructor を追加
├─ #xx Intent を値オブジェクト化
├─ #xx VO を導入
├─ #xx std::fs::write 直呼びを境界に寄せる
└─ #xx 未配線 command を配線
   … （逸脱ごとに 1 サブ Issue）
```

## 教訓

- リファクタリングの第一手は「書き直す」ではなく「理想形を定義し、そこからの逸脱を洗い出す」。逸脱が少なければ全面書き直しは不要になる。
- 逸脱の数が判断材料になる。5 箇所なら差分修正、大半が逸脱なら書き直しも選択肢、と量で決める。
- 逸脱を細かい Issue に落とすと、レビュー・巻き戻しの単位が小さくなり、途中で止めても意味のある状態が保てる。
