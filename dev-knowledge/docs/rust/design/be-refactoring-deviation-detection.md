## 問題

バックエンドをドメイン中心 + impl ベースへ「書き直す」計画を立てた。当初は
全面書き直し（rewrite）を前提に見積もっていた。

## 原因

いざ現状を調査すると、理想像に対する**逸脱（deviation）は 5 箇所だけ**だった。
既存構造の大部分はすでに理想に近く、ゼロから書き直す必要はなかった。

検出された逸脱:

1. `Config` に smart constructor がない（不変条件を型生成時に保証できていない）
2. `Intent` が raw（ドメインの意図が生の値のまま扱われている）
3. VO（Value Object）がない（プリミティブが裸で流れている）
4. 一部で `std::fs::write` を直呼びしている（I/O 境界が抽象化されていない）
5. 未配線の command がある（実装済みだが導線に繋がっていない）

「全面書き直し」という枠で見ると膨大な作業に見えるが、実体は
**局所的な 5 つのギャップ**でしかなかった。

## 解決

方針を「全面書き直し」から「**逸脱箇所を細粒度で潰す**」へ転換した。

- 理想アーキテクチャ（ドメイン中心 + impl ベース）を基準線として先に定義する。
- 現状との差分を洗い出し、**基準線からの逸脱だけ**をタスク化する。
- Epic + サブ Issue 13 件という細かい単位に分解し、1 件ずつ潰せるようにする。

### 教訓

- リファクタリングの実体は「書き直し」ではなく「**逸脱検出（deviation detection）**」で
  あることが多い。理想像を先に固定し、そこからのズレだけを対象にする。
- 見積もり前に必ず現状調査する。「書き直し」という言葉に引っ張られて作業量を過大評価
  しがち。実際のギャップは数えられる程度の有限個であることが多い。
- 逸脱を Epic + 細粒度サブ Issue に落とすと、進捗が可視化でき、並行・段階的に潰せる。

### 逸脱を潰すときの型（Rust の具体手段）

- 不変条件を持つ型には **smart constructor**（`Config::new(...) -> Result<Config, _>`）を
  用意し、生成時に検証を集約する。
- raw な値・プリミティブは **Value Object / newtype** で包み、ドメインの意図を型に載せる。
- `std::fs::write` のような I/O 直呼びは境界（trait / port）に押し出し、抽象化する。
